# AetherNarrator 优化清单全量重做设计

日期：2026-07-11  
范围：`docs/17_改进与优化清单.md` 中 A–F 工作流  
原则：清单中的“已完成”仅作为历史记录，所有项目重新按真实数据链路、自动测试和用户体验验收。

## 1. 目标与完成定义

本轮工作不是给现有函数补注释或重新勾选状态，而是逐项证明功能满足预期。每个清单项只有同时满足以下条件才可标记为完成：

1. 功能链路完整，输入、运行时状态、持久化和 UI 反馈一致。
2. 自动测试覆盖核心正常路径、失败路径和旧数据兼容。
3. 在浏览器中完成对应交互或端到端场景验证。
4. 不破坏纯前端、零后端、GitHub Pages 子路径部署和本地隐私定位。

## 2. 总体架构

### 2.1 数据所有权

- `world` 是不可被游玩过程污染的世界模板，保存世界描述、初始状态、默认知识库和默认配置。
- `save` 是一次具体游玩的完整快照，保存游戏状态、对话、聊天摘要、行为记忆、知识库副本、知识库修订状态和 AI 增强开关。
- 运行时只通过当前 session 数据修改存档内容。编辑知识库、添加记忆、更新时间或接受 AI 修订均不得直接写回世界模板。
- 新周目从世界模板深拷贝；继续游戏从存档恢复；不同存档之间不得共享可变引用。

### 2.2 旧数据兼容

- 加载世界和存档时执行幂等迁移：同一份数据重复迁移结果不变。
- 缺失字段补安全默认值，旧的世界级 `behavior_records` 在首次载入时复制到对应旧存档，原数据在确认持久化前保留。
- 迁移异常时保留原对象并回退到只读兼容路径，不清空世界或存档。
- 为存档结构增加 `schema_version`，后续迁移按版本顺序执行。

### 2.3 AI 增强总开关

- 设置项名称为“AI 增强检查”，只有开启和关闭两种状态，默认关闭。
- 关闭时：保留本地指令隔离、静态世界观校验、RAG、记忆、时间和事件引擎，不产生额外 API 调用。
- 开启时：启用生成后语义世界观裁判和周期性知识库修订建议。
- 开关保存在当前存档；世界模板可提供新周目的默认值。切换立即生效，不影响正在进行的主叙事请求。
- 所有额外调用使用独立 AbortController、超时和 session 标识，旧世界或旧回合的结果不得写入当前 session。

## 3. 分批实现设计

### 批次 0：验证基线与数据层

修复 `package.json`，建立统一命令：语法检查、模块链接检查、单元测试和浏览器烟雾测试。把现有依赖和脚本声明统一，README 中的命令必须真实可运行。

重构存档数据边界：新增存档版本、迁移器、session 快照和唯一存档选择规则。当前代码按 `worldId` 只保留一个存档，因此本轮明确维持“一世界一个当前存档”的产品语义，但数据结构不再依赖世界对象保存游玩记忆。

验收：

- `npm test` 与 `npm run verify` 均存在且通过。
- 新开、继续、重开、删除世界四条流程无共享可变数据。
- 两个存档夹具的知识库、记忆、修订计数和增强开关相互隔离。
- 旧版存档夹具可加载、保存并再次加载，关键字段无损。

### 批次 1：A 工作流——世界观守卫

将 A1–A7 合并成“输入边界—生成约束—本地后验—可选语义后验”四层管线：

1. 玩家输入以结构化段落注入，固定声明其只是游戏内行动；上传资料与玩家输入均视为不可信数据。
2. 注入检测只拦截明确的系统提示窃取、身份覆盖和协议伪造。归一化覆盖 NFKC、零宽字符、间隔符和常见全角变体；普通角色扮演、剧情命令和已知角色名不得误杀。
3. 禁用概念统一为结构化规则 `{concept, aliases, unlockTags, severity}`。活跃标签来自状态标签、物品标签和在场 NPC，规则只过滤具体违规变更，不递归删除整个合法对象。
4. 本地后验在应用状态前完成，返回可展示的违规列表；叙事命中以非阻断提示呈现，并允许用户关闭该类提示。
5. AI 增强开启时才执行 A7。裁判获得当前存档知识库而非世界默认知识库，结果绑定 turn/session；遵循原清单的非阻断设计，以 hard/soft toast 提醒，不自动回滚玩家回合。

验收场景包括：古代世界引入手机、跨时代标签解锁手机、持有带标签火器、外来 IP 专属能力、普通 NPC 拒绝对白、中文变体注入、切换世界后旧裁判结果返回。

### 批次 2：B/C 工作流——知识库与记忆

知识库检索重构为可测试的纯管线：候选召回、触发判定、递归/链接扩展、统一去重、相关性排序、token 预算裁剪。每个阶段返回诊断信息，调试面板可看到条目为何命中或被裁剪。

- B1：`always`、关键词、正则和向量触发语义明确；缺少激活键不再隐式把所有条目设为常驻，而由迁移器显式赋默认模式。
- B2：作者注记作为稳定的独立消息层，明确其插入位置和缓存影响。
- B3：知识库编辑支持完整元数据、即时校验、取消不落库、保存后只重算变更条目的向量。
- B4/B9：递归触发与图谱链接使用同一 `selectedIds`，限制深度、条数和预算；优先级是排序因子，不得绕过硬预算。
- B5/B6：AI 修订生成 diff，而不是用整份返回值覆盖知识库；玩家逐条接受、拒绝或编辑。高重要度/重复记忆可作为“晋升为 lore”候选。
- B7：知识库和修订缓冲完全属于存档。
- B8：遮罩默认开启，只遮正文；编辑时可临时揭示，不把揭示偏好跨世界泄漏。
- C1/C3：记忆面板支持查看、筛选、置顶、取消置顶和删除。`pinned` 只表示常驻，不再把 `pinned === false` 错当成“禁止检索”。
- C2：摘要采用结果优先的确定性摘要；长对话做递归合并，保留身份锚点、未完成目标和重大变化。
- C4：向量与关键词分数做可解释融合；单条向量缺失不影响其他条目；异步补算绑定当前存档。
- C5：时间线以结构化事件/高重要记忆为主，叙事回合为辅。
- C6：采用纯本地跨会话方案：支持导出/导入个人记忆包，由用户主动选择合并到当前存档，不引入服务端。

验收：小/大知识库、无向量环境、非法正则、循环图谱、超预算条目、编辑取消、修订冲突、跨存档记忆隔离、记忆包导入去重均有测试。

### 批次 3：D/E 工作流——世界模拟与时间

将世界模拟从“LLM 偶尔返回若干字段”提升为显式状态机：

- 活跃事件拥有 ID、阶段、触发条件、影响、状态和更新时间；完成事件进入历史，不只保留字符串。
- NPC 动态包含目标、当前行动、地点、可见性和最近更新时间。时段推进只刷新需要更新的 NPC。
- 世界摘要由确定性状态数据生成，展示正在变化的内容、临近截止事项和玩家可介入点。
- D1 在首页、世界详情和游戏内状态面板形成一致卖点表达；D2 保持零后端隐私说明；D3 继续使用文字等级，数值只做内部计算。

时间系统统一用绝对分钟序号作为内部时间，日期、时段、时钟和相对标签都是视图：

- E1–E6：统一配置归一化，修复当前 `clock_minutes` 与日期/时段各算一套的问题；所有推进必须单调向前。
- E8：世界级 deadline 与 goal deadline 归一到同一引擎，到点、逾期和失败只触发一次，并留下结构化事件。
- E9/E10：行动耗时推动时钟、时段和日期；跨越门槛时执行 NPC/事件门控。
- E11：季节和天气分开建模；天气是可变状态，季节由配置或日历推导。
- E12：休息经过时间引擎、deadline 检查和结构化 rest 事件记录管线；叙事使用本地确定性文本，不增加 API 成本。
- E13：设置面板编辑完整时间配置并实时预览；非法组合显示错误，不静默纠正。
- E7 不建立复杂体力经济，但把时间作为截止任务和错过事件的机会成本落实，保持文字冒险定位。

验收覆盖同日推进、跨时段、跨日、闰界/自定义历法回退、休息跨 deadline、重复失败、防倒流以及旧存档无时钟字段。

### 批次 4：F 工作流与整体体验

- reduced-motion 仅减少持续动画和打字机，不移除必要反馈；主页保持静态星空。
- 适配窄屏、触摸目标、软键盘、横竖屏和安全区；弹窗支持焦点陷阱、Esc、返回焦点和滚动锁定。
- 对知识库图谱、时间线、警告操作条和设置开关做真实浏览器验收。
- 执行 CSP、HTML 转义、相对 import、GitHub Pages 子路径和无网络降级检查。

## 4. 错误处理与可观测性

- 主叙事请求失败不得推进状态、时间、记忆或知识库。
- 辅助 AI 请求失败不影响主回合；失败原因写入调试日志，不反复 toast 干扰玩家。
- 后台任务必须携带 `{worldId, saveId, sessionEpoch, turnId}`，写入前再次校验。
- 存储写入采用序列化前校验；配额不足时提示导出备份，并保留内存状态。
- RAG 调试信息只保存条目 ID、分数和命中原因，不记录 API Key。

## 5. 测试策略

### 单元测试

覆盖迁移、状态归一化、指令检测、禁用概念、RAG 触发/预算/图谱、记忆排序、时间序号、deadline 和摘要。

### 集成测试

使用 DOM 与 fetch mock 覆盖创建世界、开始游戏、处理回合、保存、加载、重开、知识库编辑、AI 修订确认和增强开关。

### 浏览器验收

以模拟模式运行完整流程，并以受控假 API 验证流式主请求和辅助请求竞态。桌面与移动视口各执行一次关键路径。

### 完成证据

每个清单项在 `docs/17_改进与优化清单.md` 中增加“验收证据”链接或测试名称。没有证据的项目保持未验证状态，不因代码中存在同名函数而标记完成。

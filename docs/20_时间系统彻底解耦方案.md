# 时间系统彻底解耦方案（docs/20）

> 目标：让每种时间模式成为**独立的计时方案**，支持真实年/月/日（含闰年、跨年）、各世界自定义起点（不再默认 1 月 1 日）、以及大段时间跳跃（闭关三月 / 昏迷五年跳到另一个历史节点）。
> 状态：方案文档（**方案 B 已拍板，见第 12 节**）。确认步骤方案（第 13 节）后按本文档动手。

---

## 0. 你的核心诉求（复述对齐）

1. **彻底解耦**：每种时间模式单独计时，不再用"起始 day 折算公历"。
2. **审计现有世界观**：看看克苏鲁 / 后室要不要跟着改。
3. **用日历模式的新世界不再默认 1 月 1 日**：例如都市题材从"英伟达上市日 1999-01-22"起。
4. **大段时间跳跃**：
   - 修仙：今天是 2 月 3 日，闭关三个月 → 下一段剧情变成 5 月 3 日。
   - 真实世界观：今天 1999-01-22（英伟达上市），车祸变植物人，苏醒时直接跳到 2004-06-16（腾讯上市）。
5. **虚构历法（魔法历 / 星际历）**：各世界可用自己的月份表，引擎不内置任何历法假设。
6. **双世界穿梭（道诡异仙效果）**：现实 / 异界各自时间独立流逝。
7. **时间穿越（逆跳 / 分支 / 重置）**：回到过去、过一会穿回来，全程不崩。

## 1. 现有家底审计（要动哪些点）

时间系统当前**只有一条线性计数器** `current_date.day`，公历只是它的显示皮肤（`calendarLabel` 按"每月 30 天"硬折算）。这导致：
- 想显示真实日期只能把日期塞进 `day`（day:32 = 2 月 2 日），且**无法表达真实年份、闰年、跨年**。
- 无法做"闭关三个月"这类按月/按年的原生跳跃。

全项目读取 `current_date.day` 的点（约 15 处），分三类：

| 类别 | 位置 | 改造方式 |
|---|---|---|
| 显示 | theme.js:60/67/69、theme.js:80-88(formatTimeShort)、rag.js:535、render.js:735(截止)/777/828(日志) | 改用原生字段（year/month/date） |
| 记录 | save.js:81、game.js:629/674/763/859(会话历史)、game.js:515(goal failed_at) | 存原生字段 |
| 比较/推进 | time-engine.js(advanceWorldTime/collectDueDeadlines)、prompt.js:556-558(事件触发)、game.js:456-472(applyStateChanges)、llm.js:593(兜底+1天) | 走集中式 `compareCalendar` / `advanceCalendarTime` |

**两个内置世界**：
- `demo_cthulhu`：gregorian，当前 `current_date:{day:1}`，开场白"1926 年的冬天"。
- `demo_urban_legend`（后室）：`calendar_mode:"none"`，其 lore 明说"时间在这里是模糊的、没有日出日落"。
- `createDemoWorld`（用户自建世界）：默认 `calendar_mode:"day"`（第 N 天），**不会**默认成 1 月 1 日 → 诉求 3 只需约束"gregorian 模式必须带起点"。

## 2. 原有世界观是否需要调整

| 世界 | 是否要改 | 改什么 |
|---|---|---|
| **克苏鲁** | ✅ 要改 | `time_config` 加 `calendar_start:{year:1926,month:2,day:2}`（圣烛节，神话体系最对味的起点）；`current_date` 改为原生 `{year:1926,month:2,date:2,period:"morning"}`；目标截止 `day:1/3` 改为原生月/日（如 `{month:2,date:2,period:"afternoon"}` / `{month:2,date:4,period:"night"}`）。开场白"1926 年的冬天"与 2 月初深冬严丝合缝，无需改文案。 |
| **后室** | ❌ 保持 `none` | 其设定本就是"时间模糊、无日出日落"，无日历最贴合。若要强行给它真实起点反而违和。 |

**结论**：只克苏鲁需改；后室保持；新日历类世界用 `calendar_start` 自定义起点。

## 3. 目标架构：方案 A（推荐）/ 方案 B（备选）

### 方案 A：原生字段 + 隐藏序数（推荐，改动小、好审查）

**原则（关键修正：step 与 calendar_date 双轨分离）**：`current_date` 按模式携带**原生字段**用于显示与 AI；引擎内部维护两个互不相干的量——
- **`step`（剧情步，仅增）**：记录"这是第几个剧情节点"，用于事件触发记录、goal 进度、已读去重。**它只增不减，与时间轴方向无关**——无论穿越、双世界、闭关，每推进一回合 step+1。
- **`calendar_date`（真实历法日期，可任意 set / 前进 / 后退）**：各模式原生字段（gregorian 的 year/month/date、custom 的 cyear/cmonth/cdate 等），用于显示、AI 叙事、deadline 比较。
- 二者彻底解耦：step 不假装等于天数，calendar_date 可自由跳转。**比较/触发用 calendar_date 与目标日期比，且已触发的事件用"触发记录"去重（不靠序数大小），这样逆行回去不会重触发。** 作者用真实日期写世界，玩家看真实日期，AI 拿真实日期，底层 step 序数对它们不可见。

`time_config` 新增字段：
- `calendar_start`：`{year?, month, day}`（gregorian/lunar 的世界起点；period/none 模式忽略）。

`current_date` 各模式形状：
- `day` / `period` 模式：`{ day, period, clock?, relative_label? }` —— **不变**。
- `gregorian` 模式：`{ year, month, date, day, period, clock? }` —— `day` 为隐藏序数（如 1999-01-22 ≈ 72 万+，仅引擎用）。
- `lunar` 模式：`{ lYear, lMonth, lDate, day, period }`。
- `none` / `hidden` / `continuous`：`{ period, relative_label? }`。

新增 `src/calendar.js`（集中真日历运算）：
- `isLeapYear(y)`、`daysInMonth(y,m)`（处理 28/29/30/31）
- `gregorianOrdinal(y,m,d)` / `gregorianFromOrdinal(n)`（用儒略日算法，不依赖 `Date` 对象，无时区坑，支持公元 1 年起）
- `addGregorianDays(cd, n)`、`addGregorianMonths(cd, n)`（月末夹紧：1/31 + 1 月 → 2/28 或 2/29）
- `getTimeOrdinal(cd, mode)`（统一取比较用的序数）
- `formatCalendarDate(cd, mode)`（"1926年2月2日" / "农历正月初九" / "新月2·3日" / "第N天"）
- `advanceCalendarTime(currentDate, change, mode, periods)`（见第 4 节）

**为什么算"彻底解耦"**：作者、玩家、AI 三层都只接触各自模式的原生时间表示；底层 `day` 序数对它们不可见，不构成"day→公历转换"那种耦合。

### 方案 B：完全去除 `day` 序数（已拍板，详细设计）

**原则（关键：去掉"参与日历换算的隐藏 day 序数"，但保留"剧情步 step"）**：
- `current_date` 按模式**只带原生字段**，不再有任何"自纪元起的天数"序数：`gregorian:{year,month,date,period}`、`lunar:{lYear,lMonth,lDate,period}`、`custom:{cyear,cmonth_index,cdate,period}`、`day/period:{day,period}`、`none:{period,relative_label?}`。
- 新增**`step`（剧情步，仅增）**：存于 `current_date.step`（或 `S.gameState.step`）。它**只用于"第几回合"类判定**（事件触发去重、goal 进度、已读标记），**绝不参与日历换算、绝不显示、绝不假装是天数**。
- 与方案 A 的本质区别：A 的隐藏 day 序数**偷偷参与日历**（显示日期 = calendar_start + (day−1) 天），所以才有 day:32 的别扭；B 的 `calendar_date` 是**完全独立自主的真日期**，日历比较直接用原生字段按模式分派，没有任何序数中间层。

**`current_date` 各模式形状（B 版）**：
- `day` / `period`：`{ day, step, period, clock?, relative_label? }` —— day 即"第 N 天"，与 step 同义（period 模式本就只有线性天数）。
- `gregorian`：`{ year, month, date, step, period, clock? }` —— **无 day 序数**。
- `lunar`：`{ lYear, lMonth, lDate, step, period }`。
- `custom`：`{ cyear, cmonth_index, cdate, step, period }`。
- `none` / `hidden` / `continuous`：`{ step, period, relative_label? }`。

**`src/calendar.js` 的 B 版 API（无 ordinal）**：
- `isLeapYear(y)` / `daysInMonth(y,m)`（处理 28/29/30/31）。
- `addGregorian(cd, change)` / `addLunar(cd, change)` / `addCustom(cd, change, custom_calendar)`：change=`{addDays?/addMonths?/addYears?}`，按各历法月长进位（月末夹紧），**直接返回新原生 cd，不碰任何序数**。
- `compareCalendar(a, b, mode, custom_calendar?)`：**核心新函数**。返回 −1/0/1，按模式分派——gregorian 比 `(y,m,d)`、lunar 比农历、custom 比 `(cyear,cmonth,cdate)`、period 比 `day`、none 退化为比 `step`。所有"时间先后"判定统一走它。
- `formatCalendarDate(cd, mode)`（与 A 同）。
- `advanceCalendarTime(currentDate, change, mode, periods, custom_calendar?)`：B 版 = 调对应 `add*` 函数更新 `calendar_date` + **`step` 无条件 +1**（常规推进/大跳跃都算一步剧情）。

**比较/推进逻辑改造（B 比 A 多的工作量，约 15 处）**：原代码读 `current_date.day` 比大小的点，B 下改为"取 mode → 调 `compareCalendar`"：

| 位置 | A 做法（不动） | B 做法（改分派） |
|---|---|---|
| `time-engine.js` advanceWorldTime | 序数 +(天) | 调 addGregorian/addLunar/addCustom 应用到 calendar_date，step+1 |
| `time-engine.js` collectDueDeadlines | 序数 ≥ deadline | `compareCalendar(current, deadline) >= 0` |
| `prompt.js` 事件触发（getPendingEventHint） | 序数区间 | `compareCalendar` 判 dayMin/dayMax |
| `game.js` applyStateChanges 时间处理 | 序数推进 | 走 B 版 advanceCalendarTime |
| `llm.js :593` 兜底 +1 天 | 序数 +1 | 调对应 add 函数 |
| 会话历史 / 日志排序 | 序数排序 | 按 step 或 compareCalendar 排序 |
| 显示层（theme/render/rag/save 记录） | 用原生字段 | **与 A 完全相同** |

→ 可见：**B 比 A 多改的只是"比较/推进"那几处逻辑**（约 6–8 处核心 + 若干排序点），显示层 A/B 一模一样。这正是 B"更纯但更重"的来源。

**B 下扩展场景（11.1–11.4）的差异**：
- 11.1 虚构历法：`compareCalendar` 对 custom 直接比 `(cyear,cmonth,cdate)`，**最名正言顺**，比 A 的 ordinal 互转更纯。✅
- 11.2 双世界：`timelines` 各线各自 mode，比较按各线 mode 分派（与 A 同结构）。⚠️
- 11.3 时间穿越：`calendar_date` 任意 set、**无 ordinal 只增包袱**，逆跳比 A 更自然；step 仍仅增、事件去重仍靠触发记录。✅
- 11.4 编辑 UI：补起点输入框，与 A 无差异。✅

> 方案 B 功能覆盖与 A **完全等价**，且概念更干净（无隐藏序数）；代价是"比较/推进"约 6–8 处逻辑要改写成分派，回归面比 A 大。**黎总已拍板采用方案 B。**

## 4. 大段时间跳跃（核心新能力）

AI 在 `state_changes.current_date` 可返回以下指令，`advanceCalendarTime` 统一处理：

| 指令 | 含义 | 示例 |
|---|---|---|
| `{ period: "night" }` | 仅切换时段（常规） | 不变 |
| `{ addDays: 90 }` | 相对 +N 天 | 远行 90 天 |
| `{ addMonths: 3 }` | 相对 +N 月（月末夹紧） | 修仙闭关三月 |
| `{ year:2004, month:6, date:16, period:"morning" }` | 绝对跳转到该日期 | 昏迷苏醒到腾讯上市 |

规则：
- **向前跳（日历变大）一律放行**；**向后倒退（日历变小）默认放行**（已拍板允许时间倒流），由第 11.3 节的"已触发事件"策略决定逆行时的事件行为。
- 叙事跨度由 AI 描写（"你昏迷了五年，醒来已是 2004 年……"），引擎只负责日期数值。

**示例演算**：
- 修仙 `2月3日` + `addMonths:3` → `5月3日` ✅
- 都市 `1999-01-22` → 车祸 → 绝对 `{2004,6,16}` → `2004年6月16日` ✅（向前，放行）

## 11. 扩展场景（你新提的三个边界玩法 + 已触发事件四策略）

> 这三个场景会压力测试方案 B 是否健壮。结论：**方案 B 的"双轨分离"设计（step 仅增 + calendar_date 原生独立、无任何序数中间层）天然撑得住，只需在 schema / 显示层做增量扩展，不推翻比较与推进逻辑。**

### 11.1 虚构历法（魔法历 / 星际历）

**影响：无破坏性影响。** 比较器是 step（剧情步），显示与 AI 都看 `calendar_date`。只需在 schema 新增一种 `calendar_mode`：

```js
calendar_mode: "custom",
custom_calendar: {
  year_name: "星历",
  months: [ {name:"熔火月", days:35}, {name:"寒霜月", days:35}, /* ...任意月数/月长 */ ],
  // 可选：闰月规则、纪元换算
}
current_date: { cyear:3024, cmonth_index:0, cdate:3, period:"morning" }
```

- `calendar.js` 增加 `addCustom` / `compareCalendar` 的 custom 分支，按 `months[].days` 进位（星际历 13 月、魔法历月相 28 天等都表达得了）。
- AI 的 `addMonths`/`addYears` 跳跃按**该历法的月长**计算，不再是 30 天硬折算。
- step 与"历法天数"完全解耦后，星际历"一年不是 365 天"不再造成混乱——step 只是剧情步，不假装是天数。
- 闰年概念换成"该历法是否有特殊月"，由作者自定义，引擎不内置任何历法假设。

→ **本期是否纳入**：✅ 已确认纳入本期（方案 B 的"模式扩展"），与 gregorian 同构，成本低，无它则 greorian 仍唯一。

### 11.2 双世界穿梭（道诡异仙效果：现实 / 异界来回跳，各自时间独立）

**影响：中等。** 单存档单 `current_date` 不够，需升级为"多时间线"。

```js
time_config: {
  mode: "multiverse",
  timelines: {
    earth:   { calendar_mode:"gregorian", calendar_start:{year:2003,...}, current_date:{...} },
    xianxia: { calendar_mode:"lunar", current_date:{...} }
  },
  active_timeline: "earth",
  // 可选 sync_rules：异界1天=现实1年 之类的流速比，默认"各自独立流逝"
}
```

- 切换世界 = 切 `active_timeline`，**每条时间线独立保存与推进**，互不丢进度——这正是道诡异仙"两个世界时间各自流淌、主角意识来回跳"的效果。
- AI 状态只给"当前 active 时间线"的原生日期；UI 顶栏显示当前世界时间，可附另一世界"参照时间"。
- 时间流速差异（如异界 1 天 = 现实 1 年）：用可选 `sync_rules`，在本世界推进 N 步时按因子同步另一世界；道诡异仙本身各自独立，无需同步。
- 底层推进/比较（按各时间线各自的 mode 分派到 `calendar.js`）逻辑**不变**——只是从"单一 current_date"变成"字典里取 active 的那条"。

→ **本期是否纳入**：✅ 已确认纳入本期（方案 B）。`timelines` 字典 + `active_timeline` 本期实现，切换逻辑进游戏循环；各线按各自 mode 分派到 `calendar.js`，内核不变。

### 11.3 时间穿越与"已触发事件"策略（你要全兼容，这里展开分析）

**核心原则（替代"用日期大小判断事件该不该触发"）**：改用 **显式的触发记录 + 每事件的重触发策略 + 时间穿越指令携带的策略** 三者组合决定。这样四种玩法可以并存、互不打架——你确认"都要兼容"，下面给统一模型。

**四个策略**：

| 策略 | 行为 | 典型玩法 |
|---|---|---|
| **S1 不重触发（once，默认）** | 已触发事件永久进入触发记录，穿回去也不会重放 | 生日、历史性节点、一次性剧情 |
| **S2 可重复（repeatable）** | 事件带 `max_repeats` + `cooldown_steps`，到窗口且未超次数就再触发 | 每日商情、周期怪潮、按月结算 |
| **S3 时间线重置（reset）** | 穿越指令带 `reset_triggers`，回滚指定（或全部）触发记录，重走该段可重新触发 | 轮回/重开、roguelike 时间循环、"如果当初…" |
| **S4 分支隔离（branch）** | 穿回去时新建一条分支时间线，原未来作为另一分支保留；各分支独立触发记录 | 量子分叉、蝴蝶效应、平行世界（与 11.2 multiverse 同构） |

**统一数据模型（让四种都表达得了）**：
- `triggered_event_ids` 改为**按 timeline/branch 隔离**的结构：`triggered_event_ids: { <timelineId>: string[] }`（单世界也用 `main` 这一条，向后兼容）。
- 每个事件定义增加 `retrigger_policy`：
  - `"once"`（默认）→ S1；
  - `{ mode:"repeatable", max_repeats:N, cooldown_steps:M }` → S2；
  - 参与重置：出现在穿越指令的 `reset_triggers` 名单（或 `"all"`）即回滚 → S3；
  - 参与分支：父/子分支各自独立记录 → S4。
- AI 的 `state_changes.current_date` 时间穿越指令扩展为：
  ```js
  // 普通逆跳（S1 默认：保留触发记录，不重触发）
  { year:1996, month:5, date:3 }
  // 重置回放（S3）
  { year:1996, month:5, date:3, reset_triggers:["event_a","event_b"] }   // 或 "all"
  // 分支穿越（S4，与 multiverse 同构，原未来保留）
  { year:1996, month:5, date:3, branch:true, branch_label:"if-I-left" }
  ```
- **`step` 仍仅增**：分支/重置都不清空"第几步决策"计数（重置只回滚触发记录，不清 step；分支带自己的 step 起点）。这样无论怎么穿，剧情节点编号都不乱。

**本期落地（四策略全进核心引擎）**：S1/S2/S3 由"触发记录隔离 + retrigger_policy 判定 + 穿越指令显式指定"实现；S4 与 11.2 multiverse 共用"timelines 字典 + 分支"结构，本期一并实现。四策略由 **世界级默认策略 + 单事件覆盖 + 穿越指令显式指定** 三层组合决定，互不打断。

→ 演算（你要的兼容示例）：今天 1999-01-22（英伟达上市）→ 普通逆跳回 1996（S1：当年生日事件不重放，step+1）→ 带 `reset_triggers:"all"` 重走（S3：可再触发）→ 带 `branch:true` 穿回 1999 后（S4：原 1996→1999 那条未来仍作为平行分支保留）。全程 `compareCalendar` 比较器不崩。

### 11.4 对现有"时间体系编辑"UI 的影响与分档策略

**现状（已代码核查 `src/lore-ui.js:97-122` `renderTimeConfigSection`）**：创建世界初览有一个"🌐 世界时间体系"卡片，创建当次可编辑、进游戏锁定。它现在编辑：`era_label`(纪元文字)、`calendar_mode`(下拉)、`clock_mode`、`season`、`weather`、`show`。**关键缺口：它根本没有"真实起始日期"输入框**——gregorian 世界创建出来只能靠 `current_date:{day:1}` 折算成 1 月 1 日，用户/AI 无法设真实起点。这正是最初"克苏鲁开局 1 月 1 日不对味"的根因之一。

**解耦改动对它的影响 = 增强，不是破坏**：
- 方案 B 核心 + 克苏鲁起点 + 大跳跃，只需在卡片里**补一个"起始日期"输入框**（gregorian/lunar 模式显示，none/day/custom 隐藏），把 `calendar_start` 暴露给用户/AI 微调。这恰恰补上它现在的短板，让它从"只能写 era_label 文字"升级为"能设真实年月日"。
- 进游戏锁定逻辑不变（save 模式只读），不引入实时篡改风险。

**真正的复杂度风险在 11.1/11.2（custom 历法、multiverse 多世界）**：若把这俩也塞进这个轻量卡片，要填"月份表 / 多条时间线"一堆输入框，会变重、易出 bug、审查面大。

**建议分档（避免为高级玩法拖垮编辑 UI）**：
- **基础档（本期必做、低风险）**：历法下拉 + **起始日期输入框** + era/season/天气/显示。修复"默认 1 月 1 日"诉求所需，且让编辑功能更强。
- **进阶档（custom 历法、multiverse 多世界的编辑界面）本期不进编辑 UI**：留作代码级 schema（作者手写 JSON / 或日后单独做"高级时间编辑器"面板）。卡片保持轻量、稳定。**进阶点已单列到 `docs/21_时间系统进阶待办.md`。**

→ **结论：不要去掉时间体系编辑功能**。去掉等于放弃"设真实起点"能力，与原始诉求矛盾；正确做法是保留它、补起点输入框、把 heavy 高级模式挡在 UI 之外。

### 11.0 小结：三场景统一的架构支柱

| 场景 | 撑住它的设计支柱 |
|---|---|
| 虚构历法 | `calendar_mode:"custom"` + `custom_calendar` 月历表；step 与天数解耦 |
| 双世界穿梭 | `timelines` 字典 + `active_timeline`；各线独立推进 |
| 时间穿越 | `calendar_date` 可任意 set + step 仅增 + 事件"触发记录"去重（四策略全兼容） |
| 时间体系编辑 UI | 补"起始日期"输入框（基础档）；custom/multiverse 留作代码级配置不进 UI（进阶点见 docs/21） |

三者都**不推翻**方案 B 的比较/推进内核，只在 schema、显示层、切换逻辑上做增量扩展。

## 14. 时间编辑与既有文本 / 知识库的冲突处理（新增诉求）

### 14.1 问题定位（已查代码，确认当前为零防护）

你编辑了 `calendar_start`（比如克苏鲁从 1926-02-02 改到 1999-01-22），但三处**写死时间的文本**不会跟着变，造成"开局自相矛盾"：

1. **`opening_narrative`（开场白）** —— `prompt.js:276` 把它**原样纯文本**注入 system prompt，**无任何占位符机制**。克苏鲁开场白首句写死「1926 年的冬天」（`new-worlds.js:127`）。
2. **`system_prompt`（世界级 AI 主指令）** —— 克苏鲁写死「时间设定在 1920 年代」「20 年代调查」「20 年代文学气息」（`new-worlds.js:119 / 123`，共 3 处）。
3. **`lore_kb` 知识库正文** —— 条目可能提及年份/季节/时代（如"1926 年…"）；当前 schema 无时间字段，只能扫正文。

> 当前全项目**没有任何冲突检测、占位符、或自动调和机制**。若你改了时间却不改文本，AI 会把开场白里写死的旧年份当成"真相"一直讲下去。

### 14.2 设计原则

**单一时间真相（calendar_start / era_label 为权威）+ 非破坏性检测，绝不静默改写作者文本。** 作者写的东西是意图，引擎可以去"提醒"，不能去"替他改"。

### 14.3 四道机制

**机制一 · 开场白占位符（推荐，从源头消除冲突）**
- `opening_narrative` 支持**可选**占位符：`{era_label}` `{season}` `{calendar_date}` `{calendar_year}` `{calendar_month}`。
- 解析点：`prompt.js:276` 注入前 + `save.js:73` 开场记录时展开。
- 非 gregorian 模式只暴露可用 token（period→仅 `{era_label}`；none→无）。
- 作者仍可用纯文本（不强制）。采用占位符后，**时间跳跃后**下次开场/提示也自动更新（如闭关三月→自动变新日期）。
- 例：克苏鲁首句改为「现在是{era_label}的{season}。{calendar_date}，波士顿…」→ 渲染「1926 年 2 月 2 日，波士顿…」，根绝写死。

**机制二 · 编辑 / 开局时的轻量 Lint（非阻塞告警 + 一键修复）**
- 触发点：① 编辑 UI 改起始日期时；② 新游戏开局 `loadSave` 时。
- 扫描面：`opening_narrative` + `system_prompt` + `era_label` 中的硬年份/季节 token（正则 `/\b(1[89]|20)\d{2}\b/`、季节词、"如今/当代/现在/今年"）。
- 比对 `calendar_start`：年份(1600–2099) `≠ calendar_start.year`（严格相等判定，黎总 2026-07-22 拍板）或 季节 `≠ season` → 在预览/开局界面打「⚠ 时间可能冲突」徽章，列出冲突片段；**非阻塞，用户自决**。
- 不扫 lore 正文（多为世界事实，误报高，交机制三 LLM 处理）。
- **一键修复（黎总 2026-07-22 拍板）**：Lint 命中（或随时）提供两个修复按钮，均走"预览 diff → 确认才改"模态，绝不静默覆盖：
  - **重新生成开场**：LLM 按新起点生成贴合新日期的全新开场（保原调性、只改时间词）。
  - **改成占位符版**：LLM 把写死时间词替换为 `{calendar_date}`/`{era_label}`，今后时间跳跃自动跟随（更耐用，推荐优先）。
  - 按钮**始终可用**（像「重新审稿」），冲突时高亮提示；消耗一次 LLM API 调用（token 成本），按钮旁注明。

**机制三 · 复用现有 Critic 做知识库时间一致性审（LLM nuanced）**
- 扩展 `callWorldCriticLLM`（`llm.js`）审稿维度：新增「时间一致性」，把 `time_config`(era_label/calendar_start/season) 作为上下文喂给审稿 LLM，让它判断每条 lore：
  - 陈述**绝对史实**（如「英伟达 1999 年上市」）→ 不冲突（世界真相，保留）；
  - 用「现在/如今/当代/今年」等**相对当前态**措辞，但与 `calendar_start` 时代不符 → 标记冲突，提议改写；
  - 陈述**设定年代**与 `calendar_start` 明显错位且非作者刻意 → 标记。
- 复用现有 `critic.js` diff/模态流，**零新架构**。触发：世界生成后自动审稿（已存在）+ 用户改完起始日期后可手动「重新审稿」。

**机制四（可选兜底）· 权威时间覆盖**
- 每轮把 `calendar_date` + `era_label` 作为「权威当前时间」明确注入 prompt，AI 以它为真相；若开场白被忽略告警仍写死旧年份，AI 被引导以权威时间为准。兜底，不替代机制一二三。

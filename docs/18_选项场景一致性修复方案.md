# 选项场景一致性修复方案（保底选项 + AI 生成选项）

> 目标：消除两类"选项泄密/出戏"问题——
> ① 保底选项引用角色场景中根本不存在的人/地点（如后室"与警犬交流""前往 Level 2"）
> ② AI 生成选项引用角色尚未知晓的设定（如克苏鲁"《死灵之书》"）
>
> 提交人：WorkBuddy　状态：待黎总确认后动手　日期：2026-07-21

---

## 一、问题回顾与根因

| 现象 | 触发路径 | 根因 |
|---|---|---|
| 保底选项出现"与X交谈""前往Y" | `buildSmartFallbackChoices()`（game.js:512） | 它从 **lore_kb 全量设定**随机抓人物/地点名拼选项，把"世界存在过"当成"此刻在场/已知" |
| AI 选项出现《死灵之书》 | `callLLM` 同次生成 narrative+choices | lore 以"请作为事实依据/务必参考"注入，选项与叙事共享同一份知识，且无"角色认知边界"约束；A2/A7 守卫只扫 narrative 不扫 choices |

**核心缺失**：系统没有"角色当前**在场/已知**什么"的状态概念，lore 注入不分"导演视角"与"玩家视角"。

## 三、块一：保底函数改造（game.js `buildSmartFallbackChoices` 512–563）

### 现状问题代码
```js
const npcPool = kbNpcNames.length ? kbNpcNames : relNames;   // 从 lore_kb 抓人物
if (npcPool.length > 0) choices.push({ text: "与" + npc + "交谈" });      // ← 盲聊
const nearby = locations.filter(l => l !== loc);            // 从 lore_kb 抓地点
if (nearby.length > 0) choices.push({ text: "前往" + place });            // ← 盲走
```

### 改造后逻辑（伪代码）
```js
export function buildSmartFallbackChoices() {
  // 场景安全通用池：不引用任何 lore 专有名词，不假设在场人物/已知地点
  const safePool = [
    { text: "环顾四周，仔细观察当前环境",   action: "look_around" },
    { text: "检查手边能触及的物品",         action: "examine_items" },
    { text: "回想刚才发生的一切",           action: "recall" },
    { text: "试着出声呼喊，看是否有人回应", action: "call_out" },
    { text: "在原地稍作停留，整理思绪",     action: "rest" },
    { text: "让事件继续发展",               action: "continue_story" },
  ];
  // 随机抽取 3–4 个，保证每轮兜底不同
  const picked = shuffle(safePool).slice(0, 4);

  // 增强（需 L3 字段就绪才生效，默认空 → 不触发，避免盲聊/盲走）：
  const present  = (S.gameState.present_npcs || []).filter(n => n && !String(n).endsWith("_npc"));
  if (present.length)  picked.push({ text: "与" + present[0] + "交谈", action: "talk_to_" + present[0] });
  // ★ 关键：revealed_locations 必须排除角色当前所在地，绝不允许出现"前往自己脚下"
  const cur = S.gameState.current_location;
  const revealed = (S.gameState.revealed_locations || []).filter(l => l && l !== cur);
  if (revealed.length) picked.push({ text: "前往" + revealed[0], action: "go_to_" + revealed[0] });

  return picked.slice(0, 4);
}
```
- `shuffle` 用现有工具或内联 Fisher–Yates。
- **效果**：默认情况下保底选项永远只含"观察/检查/回忆/呼救/休息"——无论场景有没有人、知不知道地名都不会出戏。

## 四、块二 L1：Prompt 认知约束（data/system_prompt_template.md）

在「选项设计要求」（第 97–114 行）现有 4 条后**新增第 5、6 条**：

```
5. **选项必须限定在角色当前已知且可感知的范围内**：只能引用本轮叙事中已出现的人物/地点/物品，
   或角色此前已接触或听说的设定；不得凭空引入场景中不存在的实体（例如周围空无一人时"与他人交谈"），
   也不得提及角色尚未知晓的设定专有名词（如禁忌文献名、神话实体名），
   除非该名词在本轮叙事中已被 NPC 当场说出。
   **"前往"类选项的目标地点必须是角色当前所在地之外的其他地点，不得出现"前往自己脚下/当前所在处"的选项。**

6. **孤立/未知环境下选项应聚焦自我动作**：当角色处于独自一人或不明地点时，
   选项优先使用"观察四周、检查手边物品、回想、尝试呼救"等不依赖外部实体的动作，
   避免"去某处""找某人"之类需要前置信息才能成立的选项。
```

**同时调整「主角硬约束」（第 7–11 行）措辞**，消除与"角色本就是新手"的张力：
> 原："不得擅自将其降级为'初学者''刚入门'"
> 改为："不得削弱角色已明确设定的能力或身份；但若角色设定本身即为新手/外行，请如实反映其认知局限，不要替角色'已知'其尚未接触的设定。"

## 五、块二 L2：生成后守卫扩到 choices

### 5.1 A2 禁律守卫（game.js:697 附近）
现状只扫 `resp.narrative`。改为同时扫 `resp.choices` 文本：
```js
const choiceText = (resp.choices || []).map(c => c.text || "").join(" ");
const localViolations = findWorldviewViolations(
  resp.narrative + "\n" + choiceText,   // ← 合并扫描
  getBannedConceptRules(), getActiveConditionTags()

```

### 5.2 A7 AI 裁判（llm.js:753–758 `userContent`）
把 choices 文本附上，让裁判一并判断选项是否违和：
```js
"\n\n【待判定玩家选项】\n" + ((resp.choices||[]).map(c=>c.text).join("\n") || "（无）")
```

### 5.3（可选）轻量"未知实体"提示
若 `resp.choices` 中出现的命名实体（专名）未在本轮 `resp.narrative` 或 `behavior_records` 中出现，弹温和提示（不阻断）。复用现有 `showToast` 模式，逻辑新增于 5.1 之后。

## 六、块二 L3：认知状态追踪（较重，建议二期）

新增两个可选游戏状态字段：
- `S.gameState.present_npcs`：当前在场角色（数组）
- `S.gameState.revealed_locations`：角色已发现/已知的可达地点（数组）。**填充时必须排除当前所在地 `current_location`**，即只能放"除脚下之外、角色已知能去的地方"，否则会出现"前往自己所在处"的荒谬选项。

**填充时机**：在 `state_changes` 输出格式中增加两个可选键 `present_npcs` / `revealed_locations`，`processTurn` 应用 `state_changes` 时提取写入 `gameState`（prompt 输出格式文档需同步更新说明）。
**作用**：
- 保底函数（块一）的增强分支据此生成"与X交谈""前往Y"，且只在确有在场/已知时才出现
- L1 prompt 可引用"已揭示设定名单"作为白名单，进一步约束 AI 选项

> 说明：L3 需改 `state.js`/状态初始化、`game.js` 应用逻辑、`prompt.js` 注入、输出格式文档，工作量较大；块一+块二L1/L2 已能解决当前两个截图问题，L3 作为增强闭环建议二期实施。

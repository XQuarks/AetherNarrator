## 二、待修复问题（按优先级）

### 🔴 P0 — 数据正确性与会话隔离（影响存档安全，最高优先）
> **状态：✅ 已落地（2026-07-10）**。改动集中在 `index.html`：新增 `currentSession` / `currentAbortController` / `isGenerating` 三个全局量；`callLLM` 回传 `_sessionEpoch` / `_sessionWorldId`；`processTurn` 加串行锁 + 返回后 epoch/worldId 校验；`goHome` / `showWorldList` / `showWorldDetail` / `startGame` / `loadSave` 在导航时 `abortCurrentRequest()` + epoch 自增；`startGame(opts)` 新周目清空 `behavior_records`；世界详情弹窗区分「继续游戏 / 重新开始（覆盖确认）」。已通过 `node --check` 全量内联脚本语法校验。

#### 2.1 在途响应串世界 / 并发覆盖（会话隔离） ✅ 已修复
- **证据**：`callLLM`/`callWorldGenerationLLM` 各有 `AbortController`（4454/4502），但全局 `gameState`/`currentWorld` 在任何响应返回后都直接写入（5249 `applyStateChanges`、5275 `createOrUpdateSave`），**无 turnId/epoch 校验，导航离开时不主动 abort**。
- **修复要点**：
  1. 引入全局 `currentSession = { epoch: 0, worldId, turn }`；每次 `startGame`/`loadSave`/`submitInput` 递增 `epoch` 并存入闭包。
  2. 请求返回后先校验 `resp.epoch === currentSession.epoch && resp.worldId === currentWorld.id`，不符则丢弃（旧请求落到新世界）。
  3. 导航/切换世界/重开时 `controller.abort()` 并 `currentSession.epoch++`。
  4. 提交前加"单回合串行锁"（`isGenerating` 标志），防止并发回合按返回顺序互相覆盖。

#### 2.2 "开始游玩"静默覆盖存档 + 跨周目行为记忆不清 ✅ 已修复
- **证据（修复前）**：`createOrUpdateSave`（3186）按 `worldId` 查重并覆盖；`startGame`（3048）重置 `conversationHistory/chatHistory/chatSummary`，但**不重置世界级 `currentWorld.behavior_records`**。
- **落地**：`showWorldDetail` 打开弹窗时检测同 `worldId` 是否已有存档并动态渲染页脚——有存档则提供「继续游戏」(`loadSave`) 与「重新开始」(`startGame` + `confirm` 覆盖确认)；无存档则直接「开始游玩」。`startGame(opts={})` 默认新周目清空 `currentWorld.behavior_records` 并 `saveWorlds()` 持久化，切断跨周目行为污染。
- **修复要点**：
  1. 区分"新游戏/继续"：已有 `saves` 中存在同 `worldId` 时，弹确认「覆盖旧存档 or 另存为新周目」。
  2. 存档改用独立 `saveId`（已部分有 `id:"s"+Date.now()`），世界对象与存档对象解耦。
  3. 重开新周目前明确提示是否清空 `behavior_records`；或在 `startGame` 增加可选 `resetBehaviorRecords` 参数（默认对新周目清空，避免旧周目行为污染新周目）。

### 🟠 P1 — 核心功能正确性（RAG / 流式 / 解析 / 时间 / 上传）— ✅ 已落地（2026-07-10）

#### 2.3 向量 RAG 实际失效 — ✅ 已落地
- **证据**：`@xenova/transformers@2.17.2` 以经典 `<script src>` 加载（17 行），其返回体为 ESM（`export`），浏览器报 `Unexpected token 'export'` → `transformers` 全局永不为 `undefined` → `embeddingRetrieve` 永远进降级分支，只剩关键词检索。`loreEmbeddings` 与 `embeddingRetrieve` 的融合排序也存在 ID/内容错位风险。
- **修复要点**：
  1. 改为 `<script type="module">` 加载，或用 UMD/IIFE 构建（如 `@xenova/transformers` 提供的浏览器包路径），确保 `transformers` 全局可用。
  2. 修正关键词(7)与向量(5)融合：统一以 `snippet.id` 对齐、去重后按融合分排序取 Top8；向量不可用时纯关键词。
  3. 给向量加载失败加可见提示（而非静默降级），便于用户排查。

#### 2.4 SSE 跨分片丢 token + 超时逻辑 — ✅ 已落地
- **证据**：4537-4538 每个网络 chunk 独立 `split("\n")`，无跨 chunk 缓冲；若一条 `data:` 行被拆到两个 chunk，前半被当作残缺 JSON 跳过、后半不再以 `data: ` 开头被跳过 → 静默丢字。`clearTimeout(timeoutId)` 在 4522（收到响应头即清），仅保护连接建立，不保护流式主体。
- **修复要点**：
  1. 维持 `buffer` 跨 chunk 累积，按 `\n` 切出**完整行**才处理；未遇 `\n` 的部分留在 buffer。
  2. 收到 `[DONE]` 立即 `break` 并 `controller.abort()`。
  3. 超时改为"流式空闲超时"（如 30s 内无新 chunk 才 abort），而非收到头即清。

#### 2.5 JSON 修复不健壮 + 兜底成功推进 — ✅ 已落地
- **证据**：`tryRepairJSON`（4602）只算 `{}` 深度，未跟踪 `[]`；遇到数组内截断会补错括号。最严重：4650 在彻底失败时返回 `{"narrative":"…","choices":[]}` 被当**成功回合**处理 → 5249 `applyStateChanges`、5275 存盘、时间照常推进。
- **修复要点**：
  1. 同时跟踪 `{}` 与 `[]` 深度；字符串内 `\[` 不计数。
  2. 修复失败**不要**返回伪造成功的回合：应抛错进入 `catch`，由上层决定"重试一次 / 提示用户 / 跳过本回合（不推进时间、不覆盖存档）"。
  3. 空响应回合计：`narrative` 为空或兜底时，不调用 `applyStateChanges`、不 `createOrUpdateSave`、不推进 `current_date`。

#### 2.6 状态事务保护 — ✅ 已落地
- **证据**：`applyStateChanges(resp.state_changes)`（5249）直接 mutate 全局 `gameState`；若中途抛错，已进入的修改不会被回滚，且 `createOrUpdateSave`（5275）在之后执行——但设计上应保证"要么整轮生效、要么整轮不生效"。
- **修复要点**：
  1. 先对 `state_changes` 做 schema 校验（字段白名单 + 类型），不合法整体拒绝。
  2. 在 `deepClone(gameState)` 上应用变更，校验通过后再 `gameState = clone` 一次提交。
  3. `catch` 中**不**保存已半修改状态（当前 catch 已不 save，但应显式保证）。

#### 2.7 存档时序 — ✅ 已落地
- **证据**：`startGame` 只 `saveState()`（3066/3089）写入全局 state，不写 `saves` 列表 → 开场后存档页仍"还没有存档"；`createOrUpdateSave`（5275）早于 `finalChoices` 入 `conversationHistory`（5286），读档选项慢一轮。
- **修复要点**：
  1. `startGame` 末尾调用 `createOrUpdateSave()`，开场即生成可读存档列表项。
  2. 先确定 `finalChoices` 并写入本轮 `entry.choices`，再 `createOrUpdateSave()`，保证读档选项同步。

#### 2.8 时间系统语义 — ✅ 已落地
- **证据**：5065 `newIdx <= prevIdx` 即 `day+1`——同/早时段都加一天，违背"短操作不推进"；AI 不返回时间字段时默认推进。
- **修复要点**：
  1. 同 `period` 应视为"不跨天"（`newIdx === prevIdx` → 保持 `day`）；仅 `newIdx > prevIdx` 才 `day+1`。
  2. AI 未返回任何时间信息时，**不自动推进**（保持当前 `current_date`），并在 system prompt 强化"短操作不推进时间"。
  3. `time_mode` 修改后 `invalidateSystemPromptCache()` 已存在（3062），确认每次切换都触发，避免缓存命中旧时间配置。

#### 2.9 上传与存储配额 — ✅ 已落地
- **证据**：TXT/DOCX/EPUB 无大小/解压/条目/超时限制；仅向模型发前 8000 字，但 `source_content` 全量写 `localStorage`（约 2465），数 MB 小说即触发配额异常。
- **修复要点**：
  1. 上传前校验大小（如 ≤ 5MB），超限提示并拒绝。
  2. `localStorage` 存**截取后的参考文本**（如 ≤ 8000 字）而非全文；如需溯源可存 `fileName + hash`，原文不落盘。
  3. EPUB 解压加条目数/总大小上限，防 zip bomb。

#### 2.10 动态正则注入（ReDoS / 异常）— ✅ 已落地
- **证据**：3454 `new RegExp(escapeHtml(name), "g")`——`escapeHtml` 只转义 HTML，不转义正则元字符；`C++`、`*钥匙` 抛异常，`(a+)+$` 类可 ReDoS。
- **修复要点**：新增 `escapeRegExp(s)` 用 `\` 转义 `.*+?^${}()|[]\\` 等，替换原 `escapeHtml` 用于正则构造。

## 四、验证方式
- 每阶段用 `node --check` 保证语法；关键纯函数补单测。
- 会话隔离：开世界 A 发起请求→立即切到世界 B/首页，确认 A 的响应不写入 B。
- 存档：对同一世界"开始游玩"，确认弹覆盖确认、旧周目行为记忆按选择处理。

# 以太叙事（AetherNarrator）架构评估与方案脑暴

> 评估基准：2026-07-20 代码现状（含已落地的移动端/重复提示/force-graph 图谱、以及 prompt 方案 B 知识库移出 System 优化）。
> 规划前提：**后续优化不兼容旧存档与世界**（用户确认）。因此可自由做破坏性 schema 变更、删除冗余迁移逻辑，无需保留向后兼容。本文档为规划与讲解，未改动任何运行代码。

---

## 一、当前架构体检

### 1.1 整体定位
纯前端、零后端、可离线运行的 AI 文字冒险游戏。技术栈：原生 HTML + ES Module（无框架）+ IndexedDB + 本地内嵌中文向量模型（Xenova/bge-small-zh-v1.5）。模块化拆分彻底（26 个 `src/*.js`），工程纪律强。

### 1.2 强项（5 项）
1. **模块化与工程门禁强**：`verify-modules.cjs` 强制 import/export 完整 + 禁止裸状态引用；JSON 4 级损坏兜底（`utils.js`）；会话 epoch 失效守卫（丢弃过期响应）；`applyStateChanges` 事务回滚。
2. **RAG 设计成熟**：关键词+向量混合、bge 中文语义、触发门禁（keyword/regex/always）、递归+图谱扩展、时间线单向门禁、按 priority 的 token 预算（1600 字符 + 12 条硬上限）。
3. **提示词前缀缓存优化到位**：system 静态化 + 动态 lore 4 槽分流（before/after user、system、作者注）+ DeepSeek 前缀缓存；方案 B 后大世界 system 从几十万字符压到几千。
4. **零后端可离线 + 多世界 + UGC 便携**：`vendor/`+`models/` 本地内置，彻底摆脱 CDN；IndexedDB 多世界；世界包/记忆包导出导入。
5. **世界观守卫与防注入双层**：静态词表过滤（`worldview.js`）+ AI 语义裁判（非阻断）+ 输入归一化防注入。

### 1.3 弱点（5 项）
1. **大世界检索扩展性存疑**：`retrieve()` 每回合对全部 snippet 做关键词+向量余弦（O(n) 全扫描），时间线逐段调 Worker 算 embedding。2000+ 条无向量索引，召回成本随 n 线性增长。
2. **一致性小瑕疵**：`confirm()` 在删除存档/世界处仍原生调用（`save.js`），与已修的 restart 不一致，webview 沙箱下可能被静默吞掉。
3. **巨型模块**：`game.js`(1007)/`lore-ui.js`(1025)/`render.js`(1016)/`prompt.js`(883)/`llm.js`(803)/`utils.js`(613) 体量偏大，偏"上帝模块"。
4. **预计算向量未覆盖真实默认世界**：`lore_kb_with_embeddings.json` 只有 7 条模板向量，两个真实默认世界首次进入需运行时算向量，"预计算"名不副实。
5. **平台化（类猫箱）缺口大**：无后端/账号/同步/审核/发现目录。要做平台是架构级增量，非补丁。

### 1.4 一句话结论
工程完成度与 RAG/提示词设计成熟度明显高于平均水平；主要瓶颈在**无向量索引的大世界检索扩展性**和**完全缺失的后端/平台层**。无需推倒重来，按优先级增量优化即可。

## 二、五个维度的方案脑暴

### 2.1 知识库
- **现在**：扁平"便利贴列表"（标题+内容+触发词+类别+关系链接+向量），Obsidian 式双链已做。
- **别的路子**：
  - **图谱原生 KG**：世界建成"实体-关系-属性"知识图谱，支持推理链与矛盾检测（详见第四节）。
  - **分层本体**：先定义"什么叫一个角色/地点"（模板），再填实例，世界更整齐。
  - **Worldbook 卡片化**：社区成熟范式（SillyTavern/AI Dungeon），每张卡带触发词+插入深度，说到谁才把卡塞给 AI。
- **判断**：KG 最值得做，与现有 `links` 平滑升级；分层本体适合严丝合缝的大 IP；Worldbook 适合 UGC 自由创作。

### 2.2 逻辑处理
- **现在**：数据驱动 + AI 自由生成；有规则引擎/事件引擎/时间引擎/事务回滚，但无剧情状态机。
- **别的路子**：
  - **显式剧情图（FSM）**：提前画好场景节点+条件分支，AI 在节点内自由发挥。可控不跑飞，但灵活度下降。
  - **规则 DSL 引擎**：把写死在代码里的规则变成可配置条件（详见第四节），创作者不碰代码即可配世界。
  - **模拟优先（NPC 日程 tick）**：世界当模拟系统，NPC 每天有日程/需求/关系网，AI 只叙述结果。重投入，慎选。
- **判断**：**规则 DSL 引擎**性价比最高，最贴合"代码小白但想掌控产品"的定位。

### 2.3 提示词处理
- **现在**：固定模板+占位符，system 缓存 + 动态 lore 4 槽分流，DeepSeek 前缀缓存已优化。
- **别的路子**：
  - **分层缓存 L1–L4**：按变化频率分四层（L1 世界规则/L2 角色卡/L3 上下文/L4 对话），DeepSeek 支持多缓存断点。
  - **Few-shot 样例注入**：从高质量剧情样例库按情境检索相似好段落注入，提升文风一致性。
  - **工具调用约束输出**：用 function calling 让 AI 必须按结构返回，而非自由写 JSON 再容错修复。
- **判断**：**工具调用约束 + L2 分层缓存**低风险高收益。

### 2.4 新世界生成
- **现在**：`generateWorld()` 一次生成 schema+初始状态+开场+知识库；书>15000 字分块并发抽取再合并去重。
- **别的路子**：
  - **迭代式向导**：AI 引导多轮对话逐步补世界，体验像"和编剧搭世界"。
  - **IP 抽取 NER**：升级分块抽取，用命名实体识别自动抓角色/地点/关系→自动建图→自动生成触发词（详见第四节）。
  - **Critic 自检**：生成后用另一个 LLM 当审稿人查矛盾/漏洞/触发词冲突并自动修订（lore-revision 已有雏形）。
- **判断**：**IP 抽取 NER + Critic 自检**最贴合"上传一本书就生成可玩世界"的愿景，是差异化竞争力。

### 2.5 向量分析
- **现在**：bge-small-zh（512维）+ 余弦 + 阈值 0.30 + 关键词/向量融合，但召回是 O(n) 全库扫描 + 时间线逐段调 Worker，无索引。
- **别的路子**：
  - **ANN 索引（HNSW）**：浏览器内建向量索引，召回从"逐本翻图书馆"变"按检索卡定位"（详见第四节）。
  - **重排序（reranker）**：召回 Top-K 后用 bge-reranker 精排，相关性更高。
  - **混合检索**：BM25（关键词）+ dense（向量）融合，比自写关键词匹配更稳。
- **判断**：**ANN 索引**是"2000+ 条世界卡顿"的直接解药，优先级最高。

## 三、优化顺序（Roadmap）

> 前提：**不兼容旧存档/世界**。可自由做破坏性变更：
> - `migrations.js` 的迁移路径可清掉，直接 `LATEST_SAVE_SCHEMA_VERSION` 跳到新版本、旧数据作废；
> - snippet / save / world 的字段结构可任意调整，无需兼容 `data-foundation` 测试里的旧格式；
> - 预计算向量 JSON 可重算并直接覆盖真实默认世界。

### Phase 0 — 清理与一致性（✅ 已落地 2026-07-20，低风险）
- **目标**：消除已知小瑕疵，为后续优化扫雷。
- **改动**：统一删除存档/世界处的原生 `confirm()` 为自定义弹窗（与 restart 一致）；删除死脚本 `tools/generate_embeddings.py`（英文模型，与运行时中文模型不一致）；重算 `data/lore_kb_with_embeddings.json` 覆盖真实默认世界。
- **风险**：极低。
- **验证**：`npm run verify` 全过；浏览器冒烟无 pageerror。

### Phase 1 — 向量 ANN 索引（✅ 已落地 2026-07-20，治卡顿）
- **目标**：2000+ 条世界召回从 O(n) 全扫描改为近邻索引，消除卡顿、降未命中。
- **方案**：引入 `hnswlib-wasm`（v0.8.2，浏览器 wasm 构建，wasm 内联进 js 无需 fetch）在浏览器构建向量索引；保留关键词+向量融合与现有触发门禁/预算裁剪。
- **实际改动文件**：`src/ann-index.js`（新增，索引构建/查询/失效/兜底）、`src/rag.js`（`embeddingRetrieve` 优先 ANN、try/catch 回落 O(n)）、`src/lore-ui.js`（词条增删改后失效缓存）、`src/save.js`（切世界/读档清缓存）、`vendor/ann/`（离线库）、`index.html`（CSP 补 wasm-unsafe-eval/unsafe-eval）、`test/ann.test.js`（mock 注入测试）、`tools/ann-browser-test.mjs`（真实浏览器 2000 条重合度验证）。
- **与初版方案差异**：索引**仅在内存构建**（不持久化 IndexedDB，向量本身已持久化在 snippet 里，2000×512 建索引约 2s 且按 worldId 缓存，可接受）；未改 embedding-worker/storage/idb。
- **风险**：中（引入新依赖）→ 已缓解：① O(n) fallback 永远在，ann 失败=原行为；② 零存档/世界数据风险。
- **验证结果**：浏览器实测 2000 条——构建 ~1976ms（一次性）、单查 2.35ms、ANN topK 与暴力 topK 重合度 avgRecall=0.992 / minRecall=0.917（合格线 0.95/0.85）；`npm run verify` 全过（39 测试 + 浏览器烟雾）。

### Phase 2 — 规则 DSL 引擎（✅ 已落地 2026-07-20，让你自己配世界）
- **目标**：把硬规则从代码挪到可配置数据，创作者不碰代码即可定义条件触发。
- **方案**：世界新增 `rules:[]` 字段；`worldview.js` 加纯函数 `evaluateRules(world, gameState, text)` 解释执行 DSL（条件 `concept`/`state`/`tag`，动作 `ban`/`tag`/`ending`）；旧 `bannedConcepts` 词表仍走兼容路径（`getBannedConceptRules` 委托 `evaluateRules`）。
- **实际改动文件**：`src/worldview.js`（解释器 + `legacyBanEntry` 导出）、`src/store.js`（`rules:[]` schema + 委托）、`src/game.js`（`processTurn` 接入 tag/ending）、`src/lore-ui.js`（`openRuleEditor`/`addRule`/`deleteRule`/`saveRuleReview` 规则编辑弹窗）、`src/render.js`（世界详情加「世界规则」按钮 + 弹窗 DOM）、`index.html`（规则弹窗容器）、`styles.css`（规则卡片样式）、`test/worldview-dsl.test.js`（11 项测试）。
- **与初版方案差异**：`when` 用 `state`/`concept`/`tag` 三类具体条件（不引入未实现的 `event` 类型），`then` 用 `ban`/`tag`/`ending` 三类真实钩子（不含未实现的 `deny`）；`ending` 落点 `showGameOver()`，原 `game_over_conditions` 文案不改动不被求值。
- **风险**：低–中（DSL 解析需测试覆盖）→ 已缓解：① `evaluateRules` 纯函数 + 11 项单测；② 无 `rules` 则回退 `bannedConcepts` 零破坏；③ `severity:soft` 仅提示不阻断、`enabled` 可关、`ending` 只接真实状态判定不靠 AI 幻觉。
- **验证结果**：`npm run verify` 全过（测试 **50/50** + 浏览器烟雾 3 连跑绿，无运行时报错）；浏览器实测详情/开始/规则编辑器三流程无 pageerror；旧世界（无 `rules` 字段）打开编辑器自动补 `[]` 不崩。详见 `docs/Phase2改造方案.md`。

### Phase 3 — IP 抽取 NER + Critic 自检（高，差异化，5–8 天）**✅ 已落地（2026-07-20）**
- **目标**：上传一本书/文本，自动抽取实体→建图→生成知识库，并自动审稿修订。
- **方案**：`files.js` 抽取后接 NER（可用轻量中文 NER 或 LLM 抽取实体+关系）→ 自动建 KG + 生成 `activation_keys`/`trigger_mode`；`llm.js` 加 `callWorldCriticLLM` 做矛盾/冲突检查，`lore-revision.js` 复用做自动修订。
- **真实落地（与规划的差异，见 docs/Phase3改造方案.md）**：核对磁盘代码发现「上传文档→抽知识库」在 Plan A 已大半落地（`generateWorld` 的分块 LLM 抽取已存在），故 Phase 3 实际补的是：① **Critic 审稿人**（新增 `src/critic.js` + `llm.js` 的 `callWorldCriticLLM`，生成后自动跑、知识库面板有「🤖 审稿检查」按钮，复用 lore-revision 的 diff/确认机制，并接入 Phase 2 的 `rules` 作硬约束）；② **NER 增强**（抽取提示词加 `relations` 三元组、`mergeLoreSnippets` 保留/合并 relations、force-graph 图谱画关系边；新增「📥 从源文档补抽」按钮给已有世界 enrich）。

- **风险**：中（抽取质量需调；成本上升）。
- **验证**：`npm run verify` 全过（测试 56/56 含新增 `test/critic.test.js` 6 项、加载 18/18、浏览器烟雾全绿）；relations 合并与矛盾修订机制单测覆盖。

### Phase 4 — 知识库升级为 KG（✅ 已落地·UI 增强 + RAG 图遍历，2026-07-20）
- **目标**：在现有 `links` 基础上把知识库建模为实体-关系图，支持推理链与矛盾检测。
- **方案（实际落地 = UI 增强 v1 + RAG 图遍历 v2，黎总确认 A+A1 + 默认开启 2 跳）**：

## 四、名词大白话讲解

### ANN 索引（近似最近邻索引，Approximate Nearest Neighbor）
**现在的问题**：每回合 AI 要从 2000 条知识里找"和当前剧情相关的"，做法是"从头到尾翻一遍整个图书馆"（代码里叫 O(n) 全扫描）。条数越多越慢。
**ANN 是什么**：给图书馆"按主题分好区、建了检索卡"。AI 直接定位相关书架，不用逐本翻。HNSW 是其中一种常用算法。
**"近似"**：允许极小误差换巨大速度（比如 99% 准确但快 100 倍），对文字游戏无感。
**收益**：2000+ 条世界卡顿的直接解药，召回速度提升一个量级。

### DSL 引擎（领域特定语言，Domain-Specific Language）
**现在的问题**：世界的硬规则（比如"不能出现现代科技"）是写死在代码 `worldview.js` 里的，你（代码小白）改不了。
**DSL 是什么**：把规则变成"普通人能看懂、能填的配置语言"，像填空：
> 如果【角色 X 死亡】→ 则【触发结局 Y】
> 如果【金币 < 0】→ 则【禁止购买】
**收益**：你在界面上勾选/填写就能配世界规则，不用改 js，也不用来找我改代码。

### NER（命名实体识别，Named Entity Recognition）
**是什么**：一种 AI 技术，自动从文章里抓出"人名、地名、组织名、时间"等实体。
**用在哪**：你"上传一本书自动生成世界"时，NER 把书里的角色/地点/势力抓出来 → 自动建知识库 → 自动生成触发词。等于"AI 自动读小说并整理人物表"。
**和 Critic 的区别**：NER 是"抽信息"，Critic 是"审质量"（见下）。

### Critic（审稿人 / 评审模型）
**是什么**：让**另一个 AI** 当"挑剔的主编"，检查生成的世界/剧情有没有矛盾、漏洞、重复触发词冲突，然后自动修订。
**类比**：你写完稿让主编审一遍，挑出"第3章说他是独子、第8章冒出个弟弟"这种前后矛盾。
**现状**：你们 `lore-revision.js` 已有 AI 修订的雏形，Critic 是把"生成→审稿→修订"做成标准流水线。

### KG（知识图谱，Knowledge Graph）
**是什么**：把知识建成"点（实体）+ 线（关系）"的网，而不是现在的扁平便利贴列表。
> 哈利 —[母校]—> 霍格沃茨 —[位于]—> 英国
**好处**：
- **推理**：A 认识 B、B 效忠 C → 能推断 A 的立场；
- **矛盾检测**：若某处说"哈利不住英国"能自动报警；
- **召回增强**：说到霍格沃茨，能顺藤摸到英国、邓布利多等相关节点。
**现状**：你们 snippet 里的 `links` 字段（反向链接、关系）已经是 KG 的半成品，Phase 4 是把"半成品"补成完整的实体-关系图。

## 五、优先级总表

| 优先级 | 项目 | 价值 | 风险 | 阶段 |
|---|---|---|---|---|
| 🔥 高 | 向量 ANN 索引 | 直接治大世界卡顿+未命中 | 中 | Phase 1 |
| 🔥 高 | 规则 DSL 引擎 | 让你自己配世界，不碰码 | 低–中 | Phase 2 |
| 🔥 高 | IP 抽取 NER + Critic | 贴合"上传即生成世界"差异化 | 中 | Phase 3 |
| 中 | 知识库升级 KG | 与 links 平滑衔接，推理更强 | 中 | Phase 4 |
| 中低 | 工具调用 + L2 缓存 | 降解析失败+token | 低 | Phase 5 |
| 低/大 | 平台化（后端/账号） | 类猫箱必需 | 高 | Phase 6（独立立项） |

> 一句话：底子好，不用推翻；先动 Phase 1（治卡顿）和 Phase 2（让你配世界），这两项低风险高收益、最贴合你的定位。

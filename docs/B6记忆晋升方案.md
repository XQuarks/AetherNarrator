# B6 记忆晋升机制 · 改造方案

> 状态：✅ **已落地（2026-07-20）**——黎总确认范围（标记保留原记忆 + 阈值 importance>=4或pinned + 跟随 B5 每20轮顺带），亲口"做 B6"后落地；验证 `npm test` 78/78 + 浏览器烟雾全过。
> 前置：B5 定期回写知识库（AI 修订 + 玩家确认）已落地 ✅；B7 存档独立知识库（防跨存档污染）已落地 ✅。
> 背景：B5 当前是「软实现」——把最近行为记录当"最近事实"喂给 AI，让 AI 自行决定是否新增条目。**缺"高热度/重复记忆自动晋升为正式知识库条目"的硬机制**。本方案补这个缺口，做成 B5 回写的一个子步骤。

---

## 一、目标

把「高价值行为记录」在 B5 回写（每 20 轮 AI 审查知识库）时，**显式作为晋升候选**交给 AI，AI 若认为值得长期保留，就以新条目（addition）形式提出；玩家在既有 B5 确认台点「应用修订」即生效——完成"记忆 → 知识库"的晋升闭环。

- **不新建 UI**：完全复用 B5 的 `criticModal`/`loreReviewModal` 确认台。
- **不碰 world 出厂默认**：晋升写回 `S.activeLoreKB`（存档副本），B7 已保证不污染其他存档。
- **硬机制**：晋升候选由明确阈值筛选（不是 AI 自由发挥），可单测。

---

## 二、判定规则（默认，可后期调）

候选 = 行为记录中满足 **`importance >= 4` 或 `pinned === true`** 的条目。

- 依据：行为记录已有 `importance(1-5)` 与 `pinned` 字段（`rag.js:529` `addBehaviorRecords`）；AI 在 key_facts 时已给重要事件打高 importance（如 `game.js:479` 给 `importance:5` 的时限事件），玩家可在记忆面板手动 `pin`。
- 为什么不用"重复计数"：行为记录已按文本精确去重（`rag.js:544`），同一文本不会重复进列表，故"重复"用 `importance/pinned` 表达更稳。
- 阈值常量集中放 `src/promotion.js`，便于调：`PROMOTE_MIN_IMPORTANCE = 4`。

## 三、实现（3 处改动 + 1 个新文件）

### 1. 新增 `src/promotion.js`（纯函数，无 DOM，可 node 单测）
- `selectPromotionCandidates(records, { minImportance = PROMOTE_MIN_IMPORTANCE } = {})`
  → 返回 `records` 中 `importance >= minImportance || pinned` 的子集（深拷贝，保留 `id/text/importance/type/time/location`）。空/非法输入返回 `[]`，不崩。
- `markPromotedRecords(records, diff)`
  → 对 `diff.additions` 中 `id` 形如 `promote_<origId>` 的，把 `records` 里对应 `origId` 标记 `promoted = true`；返回**新数组**（不突变入参）+ 被晋升的 id 列表。供 `confirmLoreRevision` 调用，确保已晋升的记忆不被反复建议。

### 2. 改 `src/llm.js` 的 `callLoreRevisionLLM`（B5 回写入口）
- 在计算 `behaviorRecords` 之后，调用 `selectPromotionCandidates(S.activeBehaviorRecords)` 得到晋升候选。
- prompt 新增区块：
  ```
  晋升候选（高价值记忆，建议固化为长期知识库条目）：
  ${candidatesText || "无"}
  规则：对上述候选中你认为值得长期保留的，在 snippets 中以"新增条目"形式提出，
       且其 id 必须以 "promote_" 前缀 + 原记忆 id（如 promote_b3f2a9c1），
       content 用该记忆提炼成的正式设定；不值得保留的不要动。
  ```
- 不影响现有"最近行为记录"区块（两者并存：前者是软上下文，后者是硬晋升候选）。
- mock 模式 / API 缺失时仍返回 `null`（不阻断）。

### 3. 改 `src/lore-ui.js` 的 `confirmLoreRevision`（B5 确认入口）
- 应用 diff 后，调用 `markPromotedRecords(S.activeBehaviorRecords, diff)` 更新 `S.activeBehaviorRecords`（标记 `promoted`）。
- 随后 `createOrUpdateSave()` 已存在，会持久化 `behavior_records`（含 promoted 标记）——**无需额外存储改动**。
- 不删原记忆（保留可追溯，记忆层与知识库层职责不同）；仅标记 `promoted` 防止重复建议。

### 4. ID 约定与容错
- AI 须用 `promote_<origId>` 作为 addition id。若 AI 未遵守（如用 `nl1`），则该条目仍会作为普通新增进入知识库（晋升成功），但原记忆不标记 `promoted`——下次可能再被建议。属可接受的软劣化，prompt 已强约束前缀；不为此加复杂校验。

## 四、数据流向（晋升闭环）

```
每 20 轮触发 callLoreRevisionLLM
   ├─ 读 S.activeBehaviorRecords
   ├─ selectPromotionCandidates() → 高价值/置顶记忆
   ├─ prompt 注入「晋升候选」区块（id 前缀约束）
   └─ AI 返回 snippets（含 promote_<id> 新增）
buildLoreRevisionDiff → { updates, additions(promote_*) }
   ↓（存 S._loreRevisionBuffer，弹 toast）
玩家在确认台点「应用修订」→ confirmLoreRevision
   ├─ applyLoreRevisionDiff → 新条目进 S.activeLoreKB（存档副本，B7 隔离）
   ├─ markPromotedRecords → 原记忆标记 promoted=true
   └─ createOrUpdateSave() 持久化（知识库 + 记忆都落当前存档）
```

## 七、风险与红线
- **零回归**：只增筛选 + 标记，不删不改现有 B5 逻辑；晋升写回走既有 `applyLoreRevisionDiff` + `S.activeLoreKB`（B7 隔离）。
- **不碰 world 出厂默认**：满足。
- **不新建 UI**：满足黎总"复用 B3/B5 面板"的既有约定。
- **玩家始终有最终决定权**：晋升只是"建议候选"，必须玩家在确认台点「应用」才生效，绝不静默写入。

## 八、待确认点（黎总拍板）
1. **晋升后原记忆处理**：默认 = 标记 `promoted=true` 保留（不删，可追溯，防重复建议）。备选 = 直接移除原记忆（知识库已覆盖，更干净但丢上下文）。

3. **触发**：默认跟随 B5「每 20 轮」顺带，不独立触发（避免打扰）。

# 以太叙事 · 代码优化方案（研究版）

> 研究范围：`index.html`（单文件应用，约 244KB）全部核心模块 + `data/` 知识库 + `docs/` 设计文档 + `tools/generate_embeddings.py`。
> 目标：从**知识库/RAG 质量**、**玩法/剧情逻辑合理度**、**性能**三个维度，给出可落地的代码级优化点。
> 优先级：P0＝影响剧情一致性/正确性的逻辑缺陷；P1＝质量与体验的明显提升；P2＝性能与健壮性打磨。

---

## 一、知识库 / RAG 质量优化（核心）

### 1. [P0] RAG 融合丢失了真实匹配分数

**定位**：`retrieve()`（`index.html:3589`）+ `keywordRetrieve()`（`:3511`）+ `embeddingRetrieve()`（`:3559`）。

**问题本质**：
`keywordRetrieve` 返回的是「片段数组」，**只保留顺序、丢弃分数**；`embeddingRetrieve` 同理。到了 `retrieve()` 里：

```js
for (const s of keyword) merged.set(s.id, { snippet: s, score: 1 });   // 所有命中统一记 1 分
for (const s of embedding) {
    const existing = merged.get(s.id);
    if (existing) existing.score += 2;            // 向量命中统一 +2
    else merged.set(s.id, { snippet: s, score: 2 });
}
```

也就是说：**关键词命中 1 个还是命中 5 个，权重一样；向量相似度 0.92 还是 0.41，权重也一样**。融合排序退化成「是否出现在 Top-7 名单里」的布尔判断，余弦相似度的区分度被完全丢弃。结果是：弱相关片段和强相关片段被平等对待，注入给 AI 的知识质量下降，进而造成剧情偏离世界观。这与 `docs/04_RAG方案.md` 宣称的「关键词分 + 向量分加权合并」并不一致。

**优化方案**：让两个检索函数返回带真实分数的对象，融合时用加权求和而非固定常量。

```js
// keywordRetrieve 内部：保留加权分
const scored = kb.snippets.map(s => {
    let score = 0;
    for (const t of terms) {
        if (text.includes(t)) score += 1;
        if (kw.includes(t)) score += 1.5;
        if (title.includes(t)) score += 2;
    }
    return { snippet: s, kwScore: score };
}).filter(x => x.kwScore > 0);

// embeddingRetrieve 内部：保留余弦相似度
return scored.map(x => ({ snippet: x.snippet, embScore: x.sim }));

// retrieve 融合：真实分数加权（向量权重更高，与文档一致）
const KW_W = 1.0, EMB_W = 2.0;
// keyword 结果
for (const k of keyword) {
    const cur = merged.get(k.snippet.id) || { snippet: k.snippet, kw: 0, emb: 0 };
    cur.kw = Math.max(cur.kw, k.kwScore);   // 同片段取最高关键词分
    merged.set(k.snippet.id, cur);
}
for (const e of embedding) {
    const cur = merged.get(e.snippet.id) || { snippet: e.snippet, kw: 0, emb: 0 };
    cur.emb = Math.max(cur.emb, e.embScore);
    merged.set(e.snippet.id, cur);
}
return [...merged.values()]
    .map(x => ({ ...x.snippet, score: KW_W * x.kw + EMB_W * x.emb }))
    .sort((a, b) => b.score - a.score).slice(0, 8);
```

**收益**：检索精度显著提升，强相关世界观片段稳定入选 → 剧情更贴合设定。

### 2. [P1] 小知识库仍每轮跑 Embedding 推理（纯浪费）

**定位**：`callLLM()` → `retrieve()` → `embeddingRetrieve()`（`:3559`），以及 `buildTurnUserMessage()`（`:4002`）。

**问题本质**：当知识库全文已注入 system（`isLoreFullInSystem()` 为真，见 `:3942`）时，`buildTurnUserMessage` 会跳过动态片段注入。也就是说，**RAG 检索结果根本不会被注入 user 消息**，但每轮仍然：启动浏览器端 Transformer（MiniLM，~25MB wasm）、对玩家输入做向量编码、遍历全部片段算余弦相似度。这是每轮一次的无用重计算，在手机端尤其明显（首次推理可卡顿数秒，预热失败还会拖慢首轮）。

**优化方案**：在 `retrieve()` 里，若 `isLoreFullInSystem()` 为真，直接跳过 embedding 推理（关键词检索也只用于行为记录关联，可一起跳过）：

```js
async function retrieve(input) {
    if (isLoreFullInSystem()) {
        // 全文已在 system，动态检索无意义，仅保留行为记录召回
        const behavior = retrieveBehaviorRecords(input, 3);
        return behavior.map(b => ({
            id: "behavior_" + b.id, category: "行为记录", title: "关键事实", content: b.text
        }));
    }
    const [keyword, embedding] = await Promise.all([...]);
    ...
}
```

**收益**：小世界（绝大多数自制世界）每轮省掉一次向量推理，首字延迟明显下降。

### 3. [P0] 阈值不一致导致 12000~20000 字符区间重复注入

**定位**：`buildSystemPrompt()` 用 `LORE_FULL_THRESHOLD = 20000`（`:3739`）；`isLoreFullInSystem()` 用 `12000`（`:3946`）。

**问题本质**：知识库字符数落在 **12000 < size < 20000** 时——
- `buildSystemPrompt` 判定「全文 < 20000 → 全量注入 system」，`isCoreLoreCached = true`；
- 但 `buildTurnUserMessage` 判定 `!isLoreFullInSystem()` 为真（因为 > 12000），于是又把**非核心片段**重新注入 user 消息。

后果：这部分知识被注入两次（system 全量 + user 动态），既浪费 token，又可能让 AI 因「重复强调」而过度聚焦某几段，反而挤压其他上下文。

**优化方案**：统一为单一常量，且 `isLoreFullInSystem` 应直接复用 `buildSystemPrompt` 的判定结果（避免两处逻辑漂移）：

```js
const LORE_FULL_THRESHOLD = 12000;   // 唯一来源，两处共用
```
并把 `isLoreFullInSystem()` 改为读取 `cachedSystemPrompt` 是否为全量注入版本（或缓存一个 `isLoreFullCached` 布尔），从根本上消除两处各算各的隐患。

### 4. [P1] 行为记录的「相关度检索」结果实际被丢弃

**定位**：`retrieve()` 把行为记录作为伪片段并入 `merged`（`:3604`），但 `buildTurnUserMessage` 在 `:4004` 显式过滤掉 `behavior_` 片段，导致 RAG 召回的行为记录**从不注入** user 消息。真正注入的只有「最近 5 条」（按时间、无关相关度）。

**问题本质**：`docs/06` 描述的「按输入匹配度召回相关行为记录」机制形同虚设。当前是「固定最近 5 条」+「被丢弃的相关度结果」，既不精准又重复（同一事实可能同时出现在最近 5 条与 RAG 结果里）。

**优化方案**：把 `retrieve()` 返回的行为记录，与「最近 5 条」做**去重合并后注入**，让相关度高的旧事实也能被召回：

```js
// buildTurnUserMessage 中
const recentFacts = getRecentKeyFacts(5);
const ragFacts = retrieved
    .filter(s => String(s.id).startsWith("behavior_"))
    .map(s => s.content);
const dedupFacts = [...new Set([...ragFacts, ...recentFacts])].slice(0, 6);
```

## 二、玩法 / 剧情逻辑合理度优化（重点）

### 5. [P0] 事件触发引擎缺失——「复合触发条件」从未被引擎执行

**定位**：`system_prompt_template.md` 第 6、9 节与 `docs/03` 第 6 节都设计了**基于"时间+地点+数值+前置事件"的复合事件触发**；但 `applyStateChanges`/`processTurn` 里 **`triggered_event` 字段从未被读取**，事件片段（`category:"事件"`）只作为静态知识躺在知识库里。

**问题本质**：整个事件系统完全依赖 AI「自觉想起」。AI 不会稳定地根据"第 5 天夜里 + 在星辉 + 未完成流星雨事件"去主动推进剧情。结果是：设计文档里充满张力的**关键事件（黛玉葬花、分院仪式、流星雨之夜）几乎不会被可靠触发**，剧情沦为纯自由闲聊，玩法的"骨架"垮了。

**优化方案**：在 `processTurn` 应用状态后，加一个轻量事件评估器（纯前端、无模型调用）：

```js
// 解析事件片段里 AI 写入的触发条件（建议生成世界时让 AI 把事件结构化）
function evaluateEvents() {
    const kb = getWorldLoreKB();
    const st = gameState;
    for (const s of kb.snippets) {
        if (s.category !== "事件") continue;
        if (st.completed_events.includes(s.title)) continue;
        const cond = s.trigger || {};   // { day, period, location, minRel, prereq }
        if (cond.day && st.current_date.day < cond.day) continue;
        if (cond.period && st.current_date.period !== cond.period) continue;
        if (cond.location && st.current_location !== cond.location) continue;
        if (cond.minRel) {
            const rel = st.relationships[cond.npc] || "";
            if (!rel.includes("亲密") && !rel.includes("信任")) continue; // 简化判定
        }
        if (cond.prereq && !st.completed_events.includes(cond.prereq)) continue;
        // 命中：把事件上下文注入下一轮 user 消息，提示 AI 推进
        pendingEventHint = s;
        break;
    }
}
```

并在 `buildTurnUserMessage` 末尾注入 `# 待触发事件（请自然推进）` 区块。生成世界时（`buildWorldGenerationPrompt`）要求 AI 把事件片段补上结构化的 `trigger` 字段即可闭环。

**收益**：关键剧情节点可预期、可触发，玩法从"聊天"升级为"有结构的冒险"。

### 6. [P0] 目标系统只改 status——新增/隐藏目标转可见/deadline 失败全未实现

**定位**：`applyStateChanges` 的 `goal_updates` 处理（`:4650`）：

```js
if (changes.goal_updates) {
    for (const u of changes.goal_updates) {
        const g = s.goals.find(x => x.goal_id === u.goal_id);
        if (g) g.status = u.status || g.status;   // 只改已存在目标的 status
    }
}
```

**问题本质**：与 `docs/03` 第 8 节「目标链 / 隐藏目标 / 失败分支」严重不符：
1. **新增目标**：AI 返回一个 `goal_id` 不存在于 `s.goals` 的目标时，`find` 返回 undefined → 静默丢弃。NPC 暗示触发的隐藏任务、完成目标后生成的新目标链**全部失效**。
2. **隐藏目标转可见**：`visible` 字段从未被更新。
3. **失败分支**：`deadline` 过期后从不自动标 `failed`，设计里"deadline 过后标记 failed 并体现后果"从未发生。

**优化方案**：

```js
if (changes.goal_updates) {
    for (const u of changes.goal_updates) {
        const g = s.goals.find(x => x.goal_id === u.goal_id);
        if (g) {
            if (u.status) g.status = u.status;
            if (typeof u.visible === "boolean") g.visible = u.visible;   // 隐藏→可见
            if (u.name) g.name = u.name;
        } else if (u.goal_id && u.name) {
            s.goals.push({   // 支持目标链 / 隐藏目标揭示
                goal_id: u.goal_id, name: u.name, type: u.type || "其他",
                deadline: u.deadline || null, visible: u.visible !== false, status: u.status || "active"
            });
        }
    }
}
// 每轮检查 deadline（在 processTurn 末尾调用）
function checkGoalDeadlines() {
    const { day, period } = gameState.current_date;
    for (const g of gameState.goals) {
        if (g.status !== "active" || !g.deadline) continue;
        const over = g.deadline.day < day ||
            (g.deadline.day === day && getTimeConfig().periods.indexOf(period) >=
             getTimeConfig().periods.indexOf(g.deadline.period));
        if (over) g.status = "failed";   // 失败不等于结束，叙事体现后果
    }
}
```

**收益**：目标链、隐藏任务、失败后果真正运转，长线剧情有驱动力。

### 7. [P0] 状态写入类型污染——数字型字段覆盖文字描述

**定位**：`applyStateChanges` 的 `relationships`/`attributes`/`skills` 处理（`:4596`–`:4623`）。

**问题本质**：设计明确「属性/关系/技能用**文字描述**」。但代码对数字做了兼容：

```js
if (typeof v === "number") {
    s.relationships[k] = (typeof s.relationships[k] === "number" ? s.relationships[k] : 0) + v;
}
```

当某模型偶尔返回 `{"林黛玉": 5}`（数值型好感度）时，`s.relationships["林黛玉"]` 原本是 `"你们一见如故…"` 这种**富描述**，会被 `0 + 5 = 5`（数字）**直接覆盖**。此后所有关系上下文变成裸数字，叙事一致性崩塌。

**优化方案**：文字优先，数字仅作"提示"而非覆盖；若确为数字，转成描述或忽略：

```js
if (typeof v === "string" && v.trim() !== "") {
    s.relationships[k] = v;
} else if (typeof v === "number") {
    // 不覆盖已有描述；若想保留数值趋势，可另存 _num 供触发判断，但展示一律用文字
    const prev = s.relationships[k];
    if (typeof prev !== "string") s.relationships[k] = `好感度约 ${v}`;
    // 否则保留原文字描述，仅内部记录趋势
}
```

同时为 `processTurn` 解析阶段加一个 schema 校验：relationships/attributes/skills 期望字符串，异常值告警。

### 8. [P1] 时间回退未强制约束

**定位**：`applyStateChanges` 时间推进（`:4670`–`:4684`）+ system prompt 明确"禁止时间倒退"。

**问题本质**：引擎只做"period 不前进则 day+1"，但若 AI 把 period 写成比当前**更早**的时段（如当前 afternoon 却返回 morning），`newIdx(0) <= prevIdx(2)` 会错误地把 day+1，制造"时间倒流 + 跨天"的荒谬状态。文档禁止倒退，引擎却不校验。

**优化方案**：用「绝对时间序号」钳制，不允许回退：

```js
const tc = getTimeConfig();
if (changes.period && tc.mode === "periods") {
    const seq = d => d.day * 100 + tc.periods.indexOf(d.period);
    const prevSeq = seq(s.current_date);
    const newSeq = changes.current_date.day * 100 + tc.periods.indexOf(changes.period);
    if (newSeq < prevSeq) {
        // 非法回退：拒绝，保持原时间，仅记日志
        console.warn("AI 试图回退时间，已忽略");
    } else {
        s.current_date = { day: changes.current_date.day, period: changes.period };
    }
}
```

### 9. [P1] 兜底选项用 relationship 的 key 而非 NPC 真名

**定位**：`buildSmartFallbackChoices()`（`:4691`），`:4702` 直接取 `Object.keys(gameState.relationships)`。

**问题本质**：初始模板用的是**通用 key**（`guide_npc`/`rival_npc`/`mentor_npc`，见 `data/initial_state.json`）。于是 AI 掉链子时，兜底选项会变成「与 guide_npc 交谈」「与 rival_npc 交谈」——破坏沉浸感。

**优化方案**：优先用知识库「人物」片段的 title 作为 NPC 名，过滤掉 `_npc` 这类占位 key：

```js
const npcNames = (kb && kb.snippets ? kb.snippets.filter(s=>s.category==="人物").map(s=>s.title) : [])
    .concat(Object.keys(gameState.relationships).filter(k => !k.endsWith("_npc")));
```

## 三、性能与健壮性优化

### 10. [P2] `Intl.Segmenter` 每次分词每个 chunk 都 new 一个

**定位**：`segmentChinese()`（`:3531`），在 `for (const chunk of chunks)` 循环内 `new Intl.Segmenter("zh-CN", {granularity:"word"})`。

**问题本质**：每轮 `keywordRetrieve` + `retrieveBehaviorRecords` 都要多次构造 Segmenter。构造成本不低，移动端每轮累积开销可观。

**优化方案**：模块级单例复用：

```js
let _zhSeg = null;
function getZhSegmenter() {
    if (!_zhSeg) {
        try { _zhSeg = new Intl.Segmenter("zh-CN", { granularity: "word" }); }
        catch (e) { _zhSeg = null; }
    }
    return _zhSeg;
}
```
并把降级路径（catch）也统一到单例。

### 11. [P2] 关键事实去重过于朴素 + 自动生成模糊事实污染记忆

**定位**：`addBehaviorRecords()`（`:3626`，仅全等匹配去重）、`summarizeFactsFromChanges()`（`:3644`，产出"玩家的 courage 属性有了新的变化"这类**零信息量**事实）。

**问题本质**：
- 同一含义的不同表述（"与林黛玉关系变好了" vs "和黛玉更亲近了"）会被当成两条事实，100 条上限很快被灌满，真正重要的旧事实被挤出"最近 5 条"窗口。
- 模糊事实被无条件注入 user 消息，挤占有效上下文。

**优化方案**：
- 归一化人名/地名（黛玉↔林黛玉）后再做近义去重；
- `summarizeFactsFromChanges` 对属性/技能类变更，**仅在 AI 未提供 key_facts 时**才兜底，且把"有了新的变化"这类空话直接丢弃；
- 行为记录上限改为「保留最近 100 条 + 永久保存被引用 ≥ N 次的关键事实」。

### 12. [P2] `parseResponse` 对截断 JSON 的补全策略偏激进

**定位**：`parseResponse()`（`:4287`）+ `tryRepairJSON()`（`:4309`）。

**问题本质**：`tryRepairJSON` 在括号深度修复后，若仍失败就用正则强掏 `narrative` 字段，并用 `replace(/"/g,'\\"')` 处理——若 narrative 内含未转义的引号/反斜杠，修复后 JSON 仍非法，最终退化为 `"叙事内容获取不完整..."`，玩家看到空叙事。深层原因：`max_tokens` 在流式（`:4221`）是 6144、非流式（`:4173`）是 10240，**两份不一致**，长剧情易在非流式路径被截断。

**优化方案**：
- 统一 `max_tokens`（建议都设 8192，兼顾成本与完整性）；
- `tryRepairJSON` 在强掏 narrative 时改用更稳的「从最后一个合法 `"narrative":"` 起点截到文本末并补引号」逻辑，并对 `\`、换行做正确转义；
- 截断兜底前先尝试「从 content 末尾向前找最后一个完整 `}`」截断再解析。

## 四、优先级与实施建议

| 优先级 | 编号 | 优化点 | 类型 | 风险 |
|---|---|---|---|---|
| P0 | 5 | 事件触发引擎 | 玩法逻辑 | 中（需改世界生成 prompt） |
| P0 | 6 | 目标系统落地 | 玩法逻辑 | 低 |
| P0 | 1 | RAG 融合真实分数 | 知识库质量 | 低 |
| P0 | 3 | 阈值不一致 | 知识库/正确性 | 低 |
| P0 | 7 | 状态类型污染 | 玩法逻辑/数据 | 低 |
| P1 | 2 | 小库跳过 embedding 推理 | 性能 | 低 |
| P1 | 4 | 行为记录相关度召回 | 知识库质量 | 低 |
| P1 | 8 | 时间回退钳制 | 玩法逻辑 | 低 |
| P1 | 9 | 兜底选项 NPC 真名 | 体验 | 低 |
| P2 | 10 | Segmenter 单例 | 性能 | 低 |
| P2 | 11 | 事实去重/降噪 | 知识库质量 | 低 |
| P2 | 12 | JSON 修复/统一 max_tokens | 健壮性 | 低 |

**建议落地顺序**（贴合"先分析后修复、分阶段验证"）：

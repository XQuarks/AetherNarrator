# 以太叙事 · 功能玩法整理与竞品对比改进建议

> 日期：2026-07-10
> 对象：AetherNarrator（以太叙事）零后端 AI 互动叙事引擎
> 目的：整理现有功能玩法 → 对比主流竞品（NovelAI / SillyTavern / AI Dungeon）→ 聚焦「知识库构建」与「记忆搭建」给出可落地改造方案
> 事实来源：本项目 `src/*.js` + `docs/` 设计文档；竞品来自官方文档与社区资料（2026-07 检索）

---

## 一、功能玩法全景（现状整理）

以太叙事当前是一条完整的「创作 → 阅读 → 推进」闭环，按数据层级可拆为 6 层：

| 层 | 职责 | 关键实现 | 现状 |
|---|---|---|---|
| **① 创作层** | 生成世界观 | `generateWorld()` + 上传 TXT/DOCX/EPUB | IP/原创、自由度 1–5、源文件、主角「硬约束」 |
| **② 知识层** | 约束 AI 不编造 | `lore_kb`（8 类：规则/地点/人物/冲突/事件/势力/物品/世界观）+ RAG 混合检索 | 关键词(1×)+向量(2×)+行为记录(1.5×) 融合 **Top8** 注入 |
| **③ 状态层** | 人物与世界数值 | `gameState`：五维属性/关系/NPC活动/技能/物品/目标/死亡 | 文字描述优先，仅在「质变」时更新 |
| **④ 记忆层** | 长程不遗忘 | `behavior_records` + `chatSummary` + 关键事实注入 | 关键事实滚动窗口最近 5 + 行为记录检索 + 聊天摘要（前 2 句，上限 10） |
| **⑤ 叙事层** | 体验 | SSE 流式 + 打字机 + 自适应基调 + 三节奏模式 + 时间系统 | 物品高亮、展开/推进/加速 |
| **⑥ 世界模拟层** | 让世界「活」 | 事件引擎（复合触发）+ NPC 自主行为 + 目标链 | 时间+地点+关系+前置 复合条件主动提示 AI |

> 第 ⑥ 层（世界模拟）是本项目相对「纯对话式」竞品最独特的资产，下文会单列。

**一句话定位**：你现在已经做出了一个「能自动生成世界观 + 带状态和事件的文字冒险模拟器」，比纯聊天（SillyTavern）多了**结构化世界模拟**，比 AI Dungeon 多了**本地零后端 + 中文 RAG**。短板集中在 ② 知识层（触发粗放）和 ④ 记忆层（扁平无结构）。

## 二、竞品机制速览

### 2.1 NovelAI（Kayra / 动漫风叙事标杆）
- **Lorebook（世界书）**：每条目有 `keys`（触发关键词）+ `constant`（常驻）+ `threshold`（相关度阈值）+ `recursive`（递归连锁触发：A 注入后又满足 B 的 key 则 B 也注入）。插入位置可选 system / 故事上下文。
- **Memory**：永久常驻在上下文开头，放「最核心设定」。
- **Author's Note**：插在故事**中部**（而非开头），用来中途纠偏、控制走向——这是 NovelAI 的精髓。
- **上下文预算**：每个 Lorebook 条目有 token 上限，总预算超限按优先级裁剪。
- **摘要压缩**：长文自动递归摘要，而非简单截断。

### 2.2 SillyTavern（开源、可玩性最强）
- **World Info / Lorebook**：关键词 + **正则触发**；插入位置 5 选（系统/作者注释/对话前/对话后/滑出）；**扫描深度（scan depth）**限制「只扫最近 N 条消息」；**token 预算**；**递归连锁触发**；**优先级 + 深度策略**。
- **长期记忆插件生态**：可接向量数据库（semantic search）、mem0 类长期记忆、跨会话记忆，部分支持语义检索而非仅关键词。
- **对话压缩（compaction）**：递归摘要，分层压缩历史。

### 2.3 AI Dungeon
- **Memory / Plot Essentials**：常驻上下文开头，保证「绝不忘」。
- **Story Cards**：**关键词触发**注入（玩家/AI 提到某词才展开设定）。
- **Doctrine**：规则约束（「本世界不允许魔法」类硬约束）。
- **Memory System（2024 重构）**：AI 自动决定写/读记忆，玩家可编辑。

### 2.4 三者共同范式（关键结论）
1. **知识库都是「关键词/正则触发」**，不是「每轮无差别全量检索」。
2. **都有 token 预算 + 优先级**，而非固定 TopN。
3. **记忆都分层**（常驻核心 + 滚动情景 + 压缩历史），且**玩家可查看/编辑**。
4. **都有递归摘要**，而非「取前两句」。

## 四、知识库构建：差距与改造方案

### 4.1 现状核心问题
代码里 `rag.js::retrieve()` 每轮对**全部输入**做混合检索、融合后取 **Top8** 无条件灌进 user message 末尾（`prompt.js::buildTurnUserMessage` 的「相关知识片段」区）。问题：

1. **无差别注入 → token 浪费 + 注意力污染**：长故事里每轮都塞 8 条片段，其中多数与当前情节无关，既挤占叙事预算，又可能把无关设定「暗示」进剧情。
2. **无视插入位置**：所有知识挤在 user message 末尾，没有「中部 Author's Note」式的纠偏位。
3. **无预算/优先级**：固定 8 条，重要设定和琐碎设定同权。
4. **无扫描深度**：早期对话里出现过的词可能一直误触发。
5. **不可编辑**：AI 生成的 12 条 lore 玩家看不到、改不了，错的知识永远错。

### 4.2 改造：迁移到「Lorebook 触发范式」
给每条 lore 片段增加触发元数据（向后兼容：老存档无元数据则退化为当前全量检索）。

**片段 schema 扩展草案：**
```jsonc
{
  "id": "x1",
  "category": "规则",
  "title": "修行境界",
  "content": "凡人→练气→筑基……",
  "keywords": ["境界","修行"],
  "embedding": [/*384维*/],
  // —— 新增触发元数据 ——
  "activation_keys": ["境界","突破","修为","筑基"],   // 命中才注入（替代全量检索）
  "trigger_mode": "keyword",                          // keyword | regex | always
  "regex": null,                                       // trigger_mode=regex 时用
  "insert_at": "user_tail",                            // system | author_note | user_tail
  "priority": 5,                                        // 1-10，预算超限先砍低优先级
  "budget_tokens": 400,                                 // 本片段占用预算上限
  "scan_depth": 6,                                      // 只扫最近 6 条消息+玩家输入
  "recursive": false                                    // 注入后若又命中其他 key 则连锁
}
```

**触发逻辑（伪代码，替换当前 `retrieve`）：**
```
function retrieveTriggered(input, recentNarrative):
  window = [input] + lastN(recentNarrative, scan_depth)
  triggered = []
  for sn in loreKB.snippets:
    if sn.trigger_mode == "always": triggered.push(sn); continue
    keys = sn.trigger_mode=="regex" ? [sn.regex] : sn.activation_keys
    if any(key matches window): triggered.push(sn)
  // 预算裁剪：按 priority 降序，累加 budget_tokens 直到耗尽
  triggered.sort(by priority desc)
  selected = []; used = 0
  for sn in triggered:
    if used + sn.budget_tokens > TOKEN_BUDGET: continue
    selected.push(sn); used += sn.budget_tokens
  // 递归连锁：注入的片段若又带来新 key，再次触发（限制递归深度 ≤3）
  return selected
```

**收益**：同相关度下 token 消耗可降 40–60%，且 AI 不再被无关知识干扰——这正是 NovelAI/SillyTavern 十年沉淀的核心。

### 4.3 引入「中部注入位」（Author's Note 等价物）
当前关键事实、事件提示全在末尾。建议新增 `author_note` 区（插在 system 与最近对话之间），放：
- 事件引擎的「当前应推进的事件」提示
- 玩家手动写的「本段剧情约束」（如「接下来要揭露凶手」）

这正是 NovelAI 控制剧情走向的杀手锏，且实现成本极低（在 `buildTurnUserMessage` 里加一个拼接段）。

### 4.4 构建质量：生成后审阅 + 编辑 UI
- **生成即展示**：`generateWorld()` 完成后弹「知识库初览」面板，列出 12 条片段，玩家可**增/删/改/调优先级**。
- **质量校验**：检测重复关键词、冲突片段（如两条「规则」互相矛盾）、过于笼统（content < 30 字）并提示。
- **手动追加**：玩家随时可往 lore 里加条目（如「我自定义了一个门派」）——竞品最强粘性来源。

## 五、记忆搭建：差距与改造方案

### 5.1 现状核心问题
当前记忆有三股流，但都偏「扁平、无重要性、不可编辑」：

| 流 | 实现 | 问题 |
|---|---|---|
| 关键事实 | `buildTurnUserMessage` 注入**最近 5 条**（滚动窗口）+ 行为记录去重 ≤6 | **滚动窗口**：早期「你是催眠之王」这类设定会被新琐事挤出；主角硬约束靠隐式 gameState，不在记忆流 |
| 行为记录 | `behavior_records` 扁平 `[{id,text,createdAt}]`，上限 100，仅关键词检索 | **无结构**（无时间/地点/相关NPC/重要性/类型）；**语义检索弱**（「黛玉病了」匹配不到「黛玉咳血」）；**不可编辑** |
| 聊天摘要 | `summarizeTurn` 取叙事**前 2 句**，每 5 轮生成，`chatSummary` 上限 10 | **粗糙**：关键转折常在第三句之后（如「主角黑化」），易丢失 |

### 5.2 分层记忆模型（核心改造）
把记忆从「一个扁平 list」拆成三层，对应竞品的常驻/情景/压缩：

```
① 核心记忆 (core)      —— 常驻 system，永远注入
   · 主角硬约束、世界根本规则、初始关键关系
   · 来源：生成时写入 + 玩家 pinned
② 情景记忆 (episodic)  —— 带元数据的事件流，检索注入
   · {text, time, location, npcs[], importance, type}
   · type: event/relationship/item/discovery
③ 压缩历史 (summary)   —— 递归摘要，替代「前2句」
```

**episodic 片段 schema 草案：**
```jsonc
{
  "id": "b001",
  "text": "玩家在潇湘馆与黛玉定下葬花之约",
  "type": "event",            // event|relationship|item|discovery
  "importance": 4,            // 1-5，≥4 优先常驻/优先召回
  "pinned": false,            // 玩家置顶，永不淘汰
  "time": "第3天·傍晚",
  "location": "潇湘馆",
  "npcs": ["林黛玉"],
  "createdAt": "2026-07-10T...",
  "embedding": [/*384维，语义检索用*/]
}
```

### 5.3 重要性分级 + 置顶（pinned）
- AI 每轮产 `key_facts` 时**顺带打 importance**（提示词要求 1–5）。
- `importance≥4` 或 `pinned` 的条目进入**常驻候选池**，永不被滚动窗口淘汰（解决「主角设定被挤出」）。
- 玩家可在记忆面板手动 `pin`（类似把「我立过血誓」钉住）。

### 5.4 递归摘要替代「前 2 句」
- `summarizeTurn` 改为：**让 AI 基于「上一摘要 + 本轮叙事」生成增量摘要**（recursive），而非截取前两句。
- 成本可控：每 5 轮才调一次，且用便宜模型或本地摘要。
- 收益：长故事不丢关键转折，token 反而更省（摘要比原文短）。

### 5.5 向量记忆检索（语义）
- 行为记录/情景记忆预计算 embedding（已有 `computeEmbedding` 基础设施）。
- 召回时用**余弦相似度**而非仅关键词 → 「黛玉病了」能召回「黛玉咳血」。
- 可与 lore 向量共用 `window.transformers`，零额外后端。

### 5.6 时间线（Chronology，差异化加分项）
- 把 `type=event` 且 `importance≥3` 的片段自动汇成**时间线视图**，玩家可回看「我都经历了什么」。
- 竞品几乎没有「可读时间线」，这是把 ⑥ 世界模拟层 + 记忆层结合的创新点，值得做。

### 5.7 记忆面板（可查看/编辑/删除/置顶）
- 新增「记忆」Tab（与现有状态 Tab 并列）。
- 展示三层记忆，支持：查看全文、编辑 text、删除误记、pin 重要项、按类型/时间筛选。
- **关键价值**：玩家能纠正 AI 的「幻觉记忆」（如 AI 记成「你杀了张三」实际没杀），这是当前完全缺失、竞品标配的能力。

### 5.8 记忆注入策略升级
`buildTurnUserMessage` 调整：
```
# 前情提要        ← 递归摘要（压缩历史，替代现 chatSummary）
# 核心记忆        ← core 常驻（主角设定/根本规则），永远在
# 关键事件(务必记住) ← episodic 中 pinned + importance≥4 + 语义召回 topK
# NPC 关系        ← gameState.relationships（维持现状）
# 相关知识片段    ← 触发式 lore（见第四章）
# 当前应推进事件  ← author_note 区（提到中部）
# 当前游戏状态    ← JSON（维持现状）
# 玩家输入
```

## 六、以太叙事的差异化优势（守住并放大）

不要盲目抄竞品——你有两块竞品做不了或做得差的：

1. **世界模拟层（事件引擎 + 目标链 + NPC 活动）**：SillyTavern/NovelAI 本质是「单线对话」，没有「世界在持续推进」的感觉。你的复合触发事件 + 目标系统让世界像在**自动运行**。→ 建议强化为卖点：「会自己往前走的世界」。
2. **零后端 + 隐私 + 中文 RAG**：AI Dungeon/NovelAI 都要账号、数据上云；你纯前端 + 本地 embedding + 中文分词。→ 这是「私有 IP 模拟器」的护城河，营销上强调。
3. **文字描述优先的数值**：避免数字膨胀的机械感，体验更「文学」。保留。

**策略**：知识库/记忆向 Lorebook 范式对齐（补短板），世界模拟层继续加料（拉长处），形成「比对话器更像一个世界，比云产品更私有」的定位。

## 八、小结

- **你现在已经「能跑且好玩」**：自动世界观生成 + 状态 + 事件 + 中文 RAG，定位清晰。
- **最大两块短板都在「数据组织方式」而非「功能缺失」**：
  - 知识库：从「每轮全量 Top8」升级到「**关键词触发 + 预算 + 中部注入**」的 Lorebook 范式。
  - 记忆：从「扁平滚动窗口 + 前2句摘要 + 不可编辑」升级到「**分层 + 重要性 + 向量 + 可编辑面板**」。
- **别丢自己的长板**：世界模拟层（事件/NPC/目标）和零后端隐私是竞品没有的，应作为核心卖点放大。
- 建议从 P0 三件套（记忆面板 / lore 触发 / 递归摘要）起步，投入小、体感提升最明显。

> 下一步：若确认方向，可挑 P0 任一项进入实现（例如先落地「记忆面板 + 重要性分级」，或先落地「lore 关键词触发」）。

# docs/19 · 孤儿存档「世界已删除」友好提醒方案

> 状态：方案待确认（黎总要求先出文档再动手）
> 关联问题：玩家在旧 demo 世界（红楼梦 / 魔法学院 / 蒸汽与魔法）里留有存档，应用启动迁移把这些世界从列表删除后，存档仍在，但点进去崩溃。

---

## 一、根因分析

### 1.1 启动迁移只删「世界」不删「存档」
`src/storage.js` 的 `loadWorlds()`（约 48–66 行）每次启动都会执行一段迁移：

```js
// 迁移：删除全部旧的 demo 世界（红楼梦/魔法学院/蒸汽与魔法），注入新世界
const OLD_DEMO_IDS = ["demo_红楼梦", "demo_magic_academy", "demo_蒸汽与魔法"];
for (const oldId of OLD_DEMO_IDS) {
    if (S.worlds.some(w => w.id === oldId)) {
        S.worlds = S.worlds.filter(w => w.id !== oldId);   // ← 只动了 S.worlds
        changed = true;
    }
}
// 注入新世界（克苏鲁 / 都市传说）...
```

它**只过滤了 `S.worlds`（世界列表）**，**完全没碰 `S.saves`（存档）**。
而 `loadSaves()`（约 100–107 行）只是把 IndexedDB 里的存档原样读回：

```js
S.saves = raw;   // 不校验 worldId 是否仍存在
```

结果：旧世界被移除，但引用这些 `worldId` 的存档成了**孤儿存档**（orphan save）——名字还在存档列表里，却再也找不到对应世界。

### 1.2 崩溃点
点开孤儿存档 → `loadSave` → `prepareSessionFromSave`（`src/save.js` 约 110 行）：

```js
S.currentWorld = S.worlds.find(w => w.id === save.worldId);  // 世界没了 → undefined
```

`find` 返回 `undefined`，但后续代码毫无判空：
- `S.activeLoreKB` 因 `S.currentWorld` 为 undefined 退化为 `null`（第 115 行）
- `loadSave` 仍执行 `showScreen("gameScreen")` 并渲染日志/选项（第 132–144 行）
- 真正崩在后续回合：叙事生成时 `getWorldSchema(S.currentWorld)` 对 `null` 取值、`S.currentWorld.id` 空指针等

### 1.3 第二个隐患（顺带修）
`renderSaveDetail`（`src/render.js` 约 413 行）有这行：

```js
S.currentWorld = world || S.currentWorld; // 供导出世界等依赖 world 的逻辑兜底
```

世界不存在时，它会把**上一个仍在内存里的世界**误赋给 `S.currentWorld`，导致「导出世界」按钮导出错误的世界。这属于隐性 bug，本次一并修掉。

## 二、修复方案（4 处改动）

### 改动 A · 存档列表显示「世界已删除」徽章（`src/render.js` · `renderSaveList` 约 308–321 行）

**现在**：
```js
container.innerHTML = S.saves.map(s => {
    const isDead = s.state && s.state.is_alive === false;
    return `
    <div class="list-item save-item${isDead ? " dead-save" : ""}">
        <div class="save-info">
            <div class="item-title">${escapeHtml(s.worldName)}${isDead ? ' <span class="dead-badge">&#x2620; 已死亡</span>' : ""}</div>
            <div class="item-meta">${escapeHtml(s.progress)}<br>最后游玩：${escapeHtml(s.updatedAt)}</div>
        </div>
        <div class="save-actions">
            <button class="save-play-btn" data-action="showSaveDetail" data-id="${s.id}">继续游玩</button>
            <button class="save-del-btn" data-action="deleteSave" data-id="${s.id}">删除</button>
        </div>
    </div>
    `).join("");
```

**改为**：
```js
container.innerHTML = S.saves.map(s => {
    const isDead = s.state && s.state.is_alive === false;
    const worldMissing = !S.worlds.find(w => w.id === s.worldId);   // ★ 新增：世界是否已删除
    const titleBadges =
        (isDead ? ' <span class="dead-badge">&#x2620; 已死亡</span>' : "") +
        (worldMissing ? ' <span class="deleted-badge">&#9888; 世界已删除</span>' : "");  // ★ 新增徽章
    // ★ 孤儿存档：按钮改「查看」，点开是说明弹窗而非进游戏
    const playBtn = worldMissing
        ? `<button class="save-play-btn" data-action="showSaveDetail" data-id="${s.id}">查看</button>`
        : `<button class="save-play-btn" data-action="showSaveDetail" data-id="${s.id}">继续游玩</button>`;
    return `
    <div class="list-item save-item${isDead ? " dead-save" : ""}${worldMissing ? " missing-world" : ""}">
        <div class="save-info">
            <div class="item-title">${escapeHtml(s.worldName)}${titleBadges}</div>
            <div class="item-meta">${escapeHtml(s.progress)}<br>最后游玩：${escapeHtml(s.updatedAt)}</div>
        </div>
        <div class="save-actions">
            ${playBtn}
            <button class="save-del-btn" data-action="deleteSave" data-id="${s.id}">删除</button>
        </div>
    </div>
    `;
}).join("");
```

**效果**：孤儿存档在列表里就有红色「⚠ 世界已删除」徽章，一眼可辨；点「查看」打开的是说明弹窗，不会进游戏。

**改为**（把「世界存在 / 不存在」拆成两个分支）：
```js
export function renderSaveDetail(saveId) {
    const stored = S.saves.find(s => s.id === saveId);
    const save = stored || null;
    if (!save) { showToast("未找到该存档", "error"); return; }
    const world = S.worlds.find(w => w.id === save.worldId);
    document.getElementById("detailSaveTitle").textContent = `存档详情 · ${save.worldName}`;
    const isDead = save.state && save.state.is_alive === false;
    const body = document.getElementById("detailSaveBody");
    const footer = document.getElementById("detailSaveModalFooter");

    // ★ 存档所属世界已被删除：友好提示，禁止进入游戏（避免 currentWorld 为 null 崩溃）
    if (!world) {
        body.innerHTML = `
            <div class="form-group">
                <label>所属世界</label>
                <p style="margin:0;font-size:15px;color:var(--danger);">${escapeHtml(save.worldName)} <span class="deleted-badge">&#9888; 已删除</span></p>
            </div>
            <div class="form-group">
                <label>状态</label>
                <p style="margin:0;font-size:14px;line-height:1.6;color:var(--text-secondary);">
                    该存档对应的世界已被删除或不存在，<b>无法继续游玩</b>。<br>
                    存档数据（进度、知识库副本等）仍保留在本地。你可以删除此存档释放空间，
                    或重新创建一个同名世界后继续游玩。
                </p>
            </div>`;
        footer.innerHTML = `
            <button class="btn secondary" data-action="returnFromSaveDetail">返回</button>
            <button class="btn danger" data-action="deleteSave" data-id="${save.id}">删除该存档</button>`;  // ★ 只留 返回/删除
        showModal("saveDetailModal");
        return;
    }

    // ★ 世界存在：原逻辑不变（修掉误赋值隐患，改为直接赋值 world）
    S.currentWorld = world;
    body.innerHTML = `...（原 body 内容，保持不变）...`;
    footer.innerHTML = `
        <button class="btn secondary" data-action="returnFromSaveDetail">返回</button>
        <button class="btn secondary" data-action="editSaveLore" data-id="${save.id}">存档知识库</button>
        <button class="btn secondary" data-action="showExportWorldChoice" data-id="${save.worldId}">导出世界</button>
        <button class="btn primary" data-action="loadSave" data-id="${save.id}">继续游戏</button>`;
    showModal("saveDetailModal");
}
```

**效果**：点开孤儿存档，弹窗明确告知「世界已删除，无法游玩」，底部只给「返回 / 删除该存档」；不再渲染「继续游戏」「导出世界」按钮，杜绝误点崩溃。同时修掉 `S.currentWorld` 误赋值隐患。

### 改动 C · `loadSave` 防御性判空（`src/save.js` · `loadSave` 约 125–129 行）

**现在**：
```js
export function loadSave(saveId) {
    const stored = S.saves.find(s => s.id === saveId);
    const save = stored || null;
    if (!save) return;
    prepareSessionFromSave(save);   // ← 这里会把 currentWorld 设成 undefined 并继续
    ...
```

**改为**（在调用 `prepareSessionFromSave` 前先判空）：
```js
export function loadSave(saveId) {
    const stored = S.saves.find(s => s.id === saveId);
    const save = stored || null;
    if (!save) return;
    // ★ 防御：存档所属世界已被删除，禁止进入游戏（currentWorld 为 null 会崩溃）
    if (!S.worlds.find(w => w.id === save.worldId)) {
        showToast(`存档「${save.worldName}」所属的世界已被删除，无法继续游玩`, "warn", 3500);
        return;
    }
    prepareSessionFromSave(save);
    ...
```

> `showToast` 已在 `save.js` 顶部 import（来自 render.js），无需新增导入。此判空是兜底——正常 UI 路径下孤儿存档已无法点出「继续游戏」，但若未来有其它代码直接调 `loadSave`，也不会再崩。

## 三、影响范围与兼容性

| 项 | 说明 |
|---|---|
| 改动文件 | `src/render.js`、`src/save.js`、`styles.css` |
| 不动 | 存档数据格式、`deleteWorld` 逻辑、IndexedDB 结构、正常存档/世界的体验 |
| 旧存档兼容 | 无影响；孤儿存档只是多了一个徽章和说明弹窗 |
| git 可回滚 | 是，纯增量改动，不删文件/数据 |
| 入口覆盖 | 列表点开、详情弹窗均已覆盖；`continueLatestSave`（世界详情「继续游戏」）只在世界存在时可达，不受影响 |

**边界情况**：若用户之后用**相同 `worldId` 重新创建**了世界（如重新建一个 `demo_红楼梦`），孤儿存档会自动「复活」——因为 `S.worlds.find` 又能命中了。本方案无需额外处理，符合预期。
